3-서브프로젝트 분해 · 지금은 P1-A 헤드리스 GameFlow + 기본 AI 상세 설계

2026-06-10 brainstorming → 설계 승인 게이트 멀티에이전트 설계+적대적 검증 완료 Phase 0(룰엔진) 위에 구축

Phase 0(코어 룰엔진, Part 1+2)은 main에 머지·검증 완료(155 tests + 10만 판 무결성). 이 보고서는 Phase 1 MVP의 전체 설계 골격과, 지금 상세 계획할 P1-A(헤드리스 게임 드라이버 + 기본 AI, 순수 .NET·dotnet 테스트)를 제시해. 승인하면 설계 스펙 + P1-A 구현 계획서를 확정할게. 설계는 그라운딩→아키텍처 패널→계획 초안→적대적 검증 워크플로우로 다졌고, 검증이 잡은 핵심 결함(§6)을 모두 반영했어.

1 Phase 1 분해 — 3 서브프로젝트, 헤드리스부터

Phase 1(싱글 vs AI 풀 게임 + UI + 애니 + DI)은 커서 셋으로 나눠 위험을 낮춰. 각 단계는 독립적으로 동작/검증 가능하고, 의존은 A→B→C.

Phase 1 분해. P1-A는 헤드리스 테스트로 룰·흐름을 확정하는 "오라클", P1-B/C가 그 위에 Unity·UX를 얹는다.

2 레이어 아키텍처 (기획서 ch10 ↔ Part 2 현실 정합)

기획서 ch10.5는 "GameFlow = 별도 FSM" 레이어를 그렸지만, Part 2가 라운드 FSM을 이미 Tichu.Core.Game(GameEngine.Apply)에 구현했어. 그래서 GameFlow는 Core 위의 얇은 드라이버 + 포트(IAgent)로 축소돼(중복 FSM 없음). 의존은 항상 안쪽(Core)을 향함.

정합된 레이어. P1-A는 Tichu.GameFlow(Core만 참조)로 신설. AI는 같은 어셈블리 Agents/에 둠(단일 구현 ⇒ 별도 IAiService 포트는 YAGNI, P1-B/C에서 필요 시 도입).

문서화한 일탈(검증 통과): 기획서 ch10.5는 AI를 Core Ai/ 안에 두지만, CLAUDE.md/메모리의 "Core는 룰 전용" 원칙에 따라 AI를 GameFlow에 둠. 동기 IAgent는 기획서의 비동기 IDecisionAgent(UniTask)와 다르나, Core를 UniTask-free로 유지하려는 의도적 선택 — UniTask 래핑은 P1-B에서.

3 P1-A 핵심 — IAgent 계약 (모든 결정 9포인트 매핑)

드라이버는 "누가 사람/AI인지" 몰라도 됨 — 모든 참가자가 동일 IAgent로 결정을 반환. 6개 메서드가 룰상 발생하는 9개 결정 포인트를 모두 덮어.

// Tichu.GameFlow/Agents/IAgent.cs — 동기, Unity/UniTask 무관
public interface IAgent {
    bool           CallGrandTichu(in DecisionContext ctx);   // 큰 티츄 콜/거절
    ExchangeChoice ChooseExchange(in DecisionContext ctx);    // 좌/파트너/우 1장씩
    bool           CallTichu(in DecisionContext ctx);        // 작은 티츄(첫 플레이 전)
    TurnDecision   DecideTurn(in DecisionContext ctx);       // 리드/팔로우/패스(+마작 소원)
    Combination?   DecideBomb(in DecisionContext ctx);       // 차례 밖 폭탄, null=안 함
    int            ChooseDragonRecipient(in DecisionContext ctx); // 용 트릭 양도 상대
}
// DecisionContext = (GameState, seat) 읽기전용 뷰: MyHand / LegalMoves / CanPass / Left·Partner·Right
// 자기 손패 + 공개정보만 노출(상대 손패 비공개). RNG는 에이전트 생성자에 주입.
결정 포인트 (룰)IAgent 메서드비고
큰 티츄 콜/거절 (8장 시점)CallGrandTichu좌석별 1회, 4명 결정 시 6장 딜+교환 전이
카드 교환 (좌/파트너/우)ChooseExchange → ExchangeChoice각 방향 1장, 4명 제출 시 동시 적용
작은 티츄 콜 (첫 플레이 전)CallTichuHand.Count==14 && Call==None일 때만
리드 / 팔로우 / 패스DecideTurn → TurnDecisionLegalMoves가 이미 리드·추종·beats·소원 필터 완료
마작 소원 (마작 리드 시)TurnDecision.Wish같은 액션의 서브필드(별도 프롬프트 X)
차례 밖 폭탄 인터럽트DecideBomb유일한 off-turn 폴; 차례 좌석은 DecideTurn로

4 P1-A 드라이브 흐름 — GameDriver · MatchRunner · FlowQuery

GameDriver.RunRound가 한 라운드를, MatchRunner.RunMatch가 1000점까지 여러 라운드를 굴려. 둘 다 결정성(시드 주입)·무작위성 0. Part 2의 테스트 전용 Simulator를 프로덕션으로 승격하며 두 버그를 수정(누적점수 실제 carry-over, 1000-동점 시 속행).

매치/라운드 드라이브. Apply가 거부를 반환하면 즉시 throw — "예외 없이 완주" 자체가 불법 상태 0 증명. Simulator 대비: 누적점수 실제 carry-over + 1000-동점 속행 수정.

5 용 양도(Dragon-gift) — 유일한 Core 변경 + 검증이 잡은 치명 결함 수정

Part 2는 용 트릭 수혜자를 (winner+1)%4로 결정론적 고정해뒀어. Phase 1은 트릭 승자(사람/AI)가 상대 2명 중 선택해야 함. 이게 Phase 1이 요구하는 유일한 Core 변경(~25줄, 5파일).

CRITICAL 적대적 검증이 잡은 결함: 초안의 "일시정지"가 실제로 작동하지 않음.

CheckRoundEnd은 CollectTrick 내부가 아니라 ApplyPlay/ApplyPass에서 CollectTrick이 리턴된 뒤 호출돼. 그래서 "Turn 할당 전 return"만으로는 라운드 종료를 못 막아 → 용 트릭이 라운드를 끝내는 경우(용이 마지막 카드이거나 원-투 성립) 양도가 조용히 누락되고 (winner+1)%4 하드코드가 재발동. 점수가 가장 크게 갈리는 라운드에서 틀림.

credited = WonByDragon ? (DragonGiftRecipient ?? (winner+1)%4) : winner — ?? 폴백으로 기존 정산 테스트 전부 그린 유지. 라운드 종료 시점의 open 트릭(FinalizeOpenTrick) 용 양도도 동일 가드로 Phase=Scoring 전 해결 — 검증이 짚은 두 번째 경로.

캡처→가드→질의→재개. ②의 가드가 핵심 수정: 양도 대기 중엔 라운드 종료를 보류하고, GiveDragon이 해결한 뒤에 보류했던 종료를 실행. ?? 폴백으로 무회귀.

Core 변경 5파일: Trick(DragonGiftRecipient), GameState(pending flag + Clone/Hash, null 센티넬 인코딩), GameAction(GiveDragon), GameEngine(캡처+가드+ApplyGiveDragon), ScoreCalculator(?? 폴백). 기존 공개 시그니처 무변경.

6 적대적 검증으로 강화된 점

설계 워크플로우의 검증 단계가 잡은 이슈와 반영 결과 — 이게 ultracode 방식의 핵심 가치야.

심각도이슈반영
CRIT용 양도 "일시정지"가 실제 작동 안 함 (CheckRoundEnd 호출 위치)ApplyPlay/Pass에 Pending==null 가드 + ApplyGiveDragon에서 보류 종료 실행 (§5)
HIGH라운드 종료 open 트릭(FinalizeOpenTrick) 용 양도가 드라이버로 처리 불가Phase=Scoring 전 양도 해결 강제, 전용 테스트 추가
HIGH태스크 순서: FlowQuery(03)가 용양도(04)에 의존하는데 앞에 배치재정렬: 04(용양도 Core)를 03(FlowQuery)보다 앞으로
HIGH시나리오 테스트(용/폭탄/소원)가 NewRound 랜덤딜로는 목표 상태 도달 불가손패 직접 구성 seam(PlayPhaseTests.PlayState 패턴)을 GameDriver에 주입 — 각 시나리오 카드 명시
HIGH1000-동점 테스트를 실제 플레이로 구성 불가능순수함수 Decide(teamA,teamB,target) 추출해 단위테스트, 통합은 [Explicit] 1개
MEDAI 휴리스틱 "최저/최소/가장 싼" 선택의 객관적 기준 부재(LegalMoves 무순서)무브 정렬 헬퍼(MoveValue/Compare) 추가·단위테스트, AiAgent가 참조
MED폭탄 순서 모순 (FlowQuery 0..3 vs 드라이버 시계방향)하나로 통일: Turn+1부터 시계방향, FlowQuery가 그 순서로 반환·테스트
LOW"마지막 1명 빼고 손패 빔" 불변식이 원-투에선 거짓(2명 잔류)종료 유형별 분기 / 약한 항상참 불변식(FinishOrder 프리픽스)으로 수정
LOW소원강제 데드엔드 방어코드실제로 도달 불가(WishIsEnforced는 충족수 존재 시만 true) → "LegalMoves≠∅ ∨ CanPass" 불변식으로 테스트화
LOWComputeHash null 센티넬 / sln 폴더 네스팅recipient는 HasValue?val:ulong.MaxValue로 폴드, sln 네스팅 명시 단계 추가
범위·결정 검증 통과: P1-A는 순수 .NET 유지(UnityEngine/UniTask 없음), UI 침투 없음, 단일 Normal 휴리스틱(ISMCTS/티어/룩어헤드 없음). 잠긴 결정 3개 모두 준수, 그라운딩 사실(ScoreCalculator:74-76, 6값 GameActionKind, ComputeHash, Simulator 버그 등)도 실파일 대조로 정확 확인.
