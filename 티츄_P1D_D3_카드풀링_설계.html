D3 — 카드 풀링 + Canvas 정적/동적 분리

매 스냅샷마다 카드 뷰를 Destroy + 재생성하던 렌더 핫스팟(RenderHand / RenderBacks / RenderTrick)을, 루트별 CardChipPool(SetActive 토글 + 내용 갱신)과 정적/동적 Canvas 분리로 전환한다. 오라클(onApply=null) 경로는 비트동일로 보존한다. 멀티에이전트 적대적 검증(5차원: 오라클·호출부·Canvas 시맨틱·테스트 가능성·상태 누수)으로 두 결함을 사전 차단했다.

! 검증이 잡은 두 결함

설계 초안의 두 가정이 적대적 검증에서 거짓(holds:false)으로 판명됐다. 둘 다 구현 전에 잡았다.

결함 1 — 중첩 Canvas는 자체 GraphicRaycaster가 필요하다.
"루트 GraphicRaycaster 하나가 자식 캔버스까지 레이캐스트한다"는 가정은 거짓. 서브 캔버스는 자체 GraphicRaycaster가 없으면 그 안의 Button이 클릭을 못 받는다. _handRoot에 Canvas만 붙이면 손패 카드 클릭(ToggleCard)이 조용히 죽어 유일한 조작면이 마비된다. → _handRoot는 Canvas + GraphicRaycaster 둘 다, 트릭·뒷면은 Canvas만.
결함 2 — CardView 버튼 상태가 풀 재사용 시 샌다(잠재 버그).
_button.interactable / onClick은 Set/SetSize/SetHighlight로 초기화되지 않고 오직 SetInteractable로만 된다. 트릭·뒷면 칩은 Set+SetSize만 부른다. 손패 칩이던 인스턴스가 풀에서 트릭·뒷면으로 재사용되면 옛 카드를 캡처한 onClick 리스너가 살아남아 뒷면 클릭이 엉뚱한 카드를 선택한다. (지금은 풀이 없어 잠복.) → 루트별 토폴로지 + Set() 버튼 중립화로 이중 차단.
나머지 3차원은 holds:true. 오라클은 뷰와 구조적으로 분리(Card=readonly struct 값복사, 방어적 List 복사). 렌더 호출부 8곳·ClearChildren 7곳 전수 확인, 동치 보존. EditMode 테스트는 기존 패턴(CardViewTests·ITableViewContractTests)으로 가능.

1 CardChipPool · 루트별 풀

핫스팟은 RenderHand/RenderBacks/RenderTrick이 매 스냅샷마다 자식을 Destroy + Instantiate하는 것(GC 스파이크 · 캔버스 리빌드). 풀은 칩을 파괴하지 않고 SetActive 토글 + 내용 재설정으로 재사용한다.

추측이 아니라 측정된 핫스팟 대응. 풀링 경로에서 Destroy/Instantiate를 완전히 제거한다.

토폴로지 — 루트별 풀 5개

칩이 제 루트에 머물러 SetParent(레이아웃 dirty) 없이 재사용된다. 좌석0(=나)은 뒷면이 없어 뒷면 풀은 3개.

API 형태

// CardView 전용(범용 제네릭은 단일 타입이라 YAGNI). 루트당 1 인스턴스.
sealed class CardChipPool {
  CardChipPool(RectTransform root, CardView prefab) { … }
  void Begin();                 // 커서 0
  CardView Next();              // 비활성 칩 재사용; 없으면 Instantiate+추가; SetActive(true)
  void End();                   // 커서 이후 칩 SetActive(false)

  // 테스트 시임(read-only) — Destroy 프레임 지연 탓에 childCount로는 단언 불가
  int CreatedCount { get; }     // 총 Instantiate 횟수
  int ActiveCount  { get; }
  int FreeCount    { get; }
}

소비자(RenderHand 등)는 Begin() 후 카드마다 var cv = pool.Next(); 받아 기존 Set/SetSize/SetHighlight/SetInteractable를 그대로 호출하고 끝에 End(). 커서 순서 = 형제 순서 = 정렬(OrderBy(SortKey)) 순서라 sibling 재정렬 불필요.

2 Canvas 정적/동적 분리

단일 Canvas에선 어떤 Graphic이 dirty돼도 캔버스 전체가 리빌드된다. 자주 바뀌는 동적 루트(손패·트릭·뒷면)를 서브 캔버스로 격리해, 카드 탭 시 손패 서브캔버스만 리빌드되고 점수판·프로필 정적 캔버스는 안 건드리게 한다. 이 이득은 풀링과 결합돼야 실현된다(풀링이 destroy/instantiate churn을 없애므로).

손패만 GraphicRaycaster 추가(필수). 트릭·뒷면은 Button이 없어 Canvas만. overrideSorting은 기본값(false) 유지.

이 섹션은 선택적으로 미룰 수 있다. 풀링만으로도 GC·instantiate 비용의 큰 부분이 사라진다. Canvas 분할은 리배칭 격리라는 추가 이득이지만, placeholder 뷰가 PrefabTableView로 교체되면 재정리 대상이다. 분할을 D6(실측) 이후로 미루는 선택도 유효 — 의사결정 항목.

3 CardView 강화 (결함 2 방어)

루트별 토폴로지가 이미 교차 재사용(손패↔트릭/뒷면)을 막지만, 소비자 규율과 무관하게 안전을 보장하도록 CardView를 소폭 강화한다(이중 차단).

  • Set() 버튼 중립화: if (_button != null) { _button.onClick.RemoveAllListeners(); _button.interactable = false; }. Set+SetSize만 부르는 트릭·뒷면 칩이 풀 이력과 무관하게 깨끗해진다.
  • HasClickListener 플래그(거동 테스트용): UnityEvent의 런타임 AddListener 클로저는 public API로 셀 수 없어, SetInteractable에서 set / Set() 중립화에서 clear하는 bool 플래그로 리스너 상태를 단언 가능하게 한다.
  • 오해 소지 주석 수정: "풀 재사용 안전을 위해…"는 소비자가 SetInteractable을 부를 때만 성립 → 버튼 리셋을 Set()로 옮겼음을 반영.
_face.sprite / _label.text 잔존은 해당 요소가 enabled=false라 시각 누수가 없다 → YAGNI, 손대지 않는다(검증 dim5 확인).

4 오라클 보존 불변식

풀링은 순수 뷰 레이어 변경이다. Card는 readonly struct(값복사)이고 ApplySnapshot이 손패를 방어적 복사(TableViewModel.cs:132)하므로, 풀이 게임 상태로 샐 수 없다(검증 dim1 확인).

⛔ 건드리지 않는다 (D3 불변식)

  • AsyncGameDriver · onApply 콜백 배선 — onApply=null 경로 비트동일
  • TableViewModel.ApplySnapshot 및 cs:132의 방어적 new List<Card>(...) 복사 ("최적화" 명목으로도 제거 금지 — 이게 aliasing을 막는다)
  • RenderTrick이 읽는 trick.Top.Cards 등 읽기전용 컬렉션 — in-place 정렬/변형 금지

변경 범위는 RenderHand/RenderBacks/RenderTrick 내부의 GameObject 수명뿐이다. 회귀 게이트: 기존 오라클 교차검증 테스트(동기 GameDriver vs 비동기, 다중 시드 ComputeHash)를 D3 전후로 그린 확인.

5 검증 · 테스트 (EditMode)

실측 수치(60fps/GC/드로콜)는 D6 실기기 게이트의 몫. D3는 동치 + 풀 불변식 + 구조를 EditMode로 못박는다. 모두 기존 패턴(new GameObject + AddComponent<CardView>, Object.DestroyImmediate teardown, Editor-only asmdef) 준수.

테스트단언비고
풀 불변식N개 빌드→CreatedCount==N; 축소(M<N)→여전히 N(신규 할당 0); 재확장→여전히 N. release된 칩 activeSelf==false.childCount 금지(Destroy 프레임 지연) → 풀 카운터로
리스너 누수콜백 arm→release→다른 카드로 re-Get(재무장 X)→onClick 호출→옛 콜백 안 불림.거동 단언(또는 HasClickListener)
