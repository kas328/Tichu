1. 왜 필요했나 — 관측된 증상

로드맵 검증에서 확인된 근본 원인: PIMC 의 라이브 수는 EV 탐색이 고른다. 그런데 순수 EV 는 전략 융합 + 롤아웃 천장 탓에 “아웃 임박 상대를 지금 막아야 한다”는 판단을 라이브 수에 직접 반영하지 못한다(롤아웃에만 간접 반영). 그래서 휴리스틱 AiAgent 는 이미 막을 줄 아는데도, PIMC 좌석은 EV 평균이 흐릿해 그냥 패스해버린다.

2. 구현 — 파트너-밟기 가드의 구조적 쌍둥이

이미 검증된 PartnerOvertakeMove(파트너가 Top 일 때 밟기 가드)와 정확히 대칭인 신규 메서드를 만들고, PimcAgent 의 파트너 가드 바로 뒤에 배선했다. 기존 DecideFollow 로직은 손대지 않았다(수술적).

  • PolicyConfig.cs — UseOpponentThreatBlock 플래그(기본 false) 추가
  • PimcAgent.cs — 파트너 가드 직후 상대-Top 게이트 배선(Seating.TeamOf 상대 판정)
  • PimcAgentTests.cs — 통합 테스트 +3 (ON 봉쇄 2종 · OFF 게이팅 1종)
  • AiAgentTests.cs (Unity+core) — 순수 메서드 테스트 +4 (규칙①·②·구조보존·무위협)

4. 적용 — PIMC 티어 승격 + 비-회귀 벤치

Normal·Hard·Expert 승격(D1 ON). 사용자 결정(A): 체감 확인을 우선해 PIMC 전 티어에서 D1 을 켰다. D1 은 난이도 차별 축이 아니라 관측된 나쁜-수를 고치는 정확성 바닥이라, Normal 만 켜면 Hard 가 같은 증상을 보이는 티어 역전이 생기므로 세 티어 모두 ON(Easy 는 Worlds=0 라 가드 미도달·무관). 회귀 위험은 낮다 — 이미 검증된 DecideFollow 로직을 좁은 위협 조건에서만 재사용하기 때문. 비-회귀 확인용 PimcBench ON/OFF 는 백그라운드로 진행 중.

저장소의 “강도 주장은 격리 벤치 필수” 규율을 존중하되, D1 의 실제 검증은 플레이테스트(체감 증상 제거)이며 벤치는 점수 비-회귀를 사후 확인한다(로드맵도 점수 이득은 “소폭”으로 예측). 승격은 TDD(UseOpponentThreatBlock_on_for_pimc_tiers RED→GREEN)로 반영, 재검증 64/64 그린.
