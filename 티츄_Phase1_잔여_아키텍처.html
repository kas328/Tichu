Tichu · Architecture Design Report

Phase 1 잔여 아키텍처 — P1-C 앱 셸·DI + P1-D 비주얼·손맛

현재 구현 현황(기획서 대조)을 점검하고, MVP 완료 기준(DoD)까지 남은 두 묶음 — 앱 흐름/메뉴/DI(P1-C)와 실제 아트/DoTween 애니/사운드/모바일 가로·60fps(P1-D) — 의 실행 아키텍처를 정의한다. 멀티에이전트 설계 패널(3안)→심사(만장일치)→종합→적대적 검증으로 도출.

1 구현 현황 점검 (기획서 ↔ 코드)

기획서는 Phase 0 → 1(MVP) → 2(AI) → 3(온라인) → 4(BM) 순차 게이트 구조다. 커밋·소스·패키지를 교차 검증한 결과, "테스트 가능한 정확한 코어 + 기능적으로 동작하는 한 판"은 완성, 기획서가 말한 "재미·손맛이 검증되는 제품"까지는 앱 셸과 비주얼 레이어가 남았다.

로드맵상 현재 위치 — Phase 1 내부에서 P1-A/P1-B 완료, P1-C·P1-D가 G1 게이트까지의 잔여.

2 다음 계획 범위 정의

사용자 확정 범위 = Phase 1 잔여 전체. 기획서 Phase 1 DoD(한 판 1000점 완주 · 중저가 안드로이드 60fps · 손맛/가독성 합격)와 5장 사이트맵을 충족하는 두 묶음으로 분해한다.

P1-C 앱 셸 · 흐름 · DI

  • 인트로 → 메인 허브 → 모드 선택 → 인게임 → 결과 → 메인 복귀 흐름
  • 게임 방법 / 설정(볼륨) / 일시정지(재개·메인복귀) / 재시작
  • 랭킹전·친구방 버튼 존재하되 Phase 3 스텁, AI전만 실 경로
  • VContainer DI 도입(앱-수명 서비스만)
  • 매치 종료(목표 점수) 개념 도입 → 결과 화면 라우팅

P1-D 비주얼 · 손맛

  • 런타임 빌드 placeholder → 프리팹 + 실제 카드 스프라이트·아바타
  • DoTween 딜 / 플레이 / 회수 / 하이라이트 / 선 표시 애니
  • 사운드 훅(SFX/BGM)
  • 모바일 가로(landscape) 고정 · CanvasScaler 재설계 · SafeArea
  • 중저가 안드로이드 60fps · 카드 풀링 · 드로콜/GC 예산
두 묶음은 독립 PR로 분리 가능하도록 설계한다. P1-C 없이도 RoundBootstrap.Begin() 직접 호출로 인게임 검증이 가능하고, P1-D는 ITableView 인터페이스 뒤에서 현행 뷰와 교체된다(§6).

4 목표 아키텍처 개요 & 불변식

핵심 철학: "현 구조는 이미 깨끗한 포트/어댑터 경계(IDecisionAgent·IHumanInputPort·onApply)를 갖췄다." 따라서 P1-C/P1-D는 새 레이어를 쌓는 일이 아니라 — 단일 진입점을 셸로 일반화하고, 런타임 빌드 UI를 프리팹으로 교체하고, onApply 스냅샷에 트윈 큐를 한 겹 끼우는 일이다.

⛔ 신성불가침 불변식 (한 줄도 건드리지 않는다)

  • 오라클 비트동일성 — AsyncGameDriver.RunRoundAsync(…, onApply=null) 경로는 동기 GameDriver와 비트 동일해야 한다(_onApply?.Invoke가 null-가드, AsyncGameDriver.cs:173). 트윈·사운드는 이 경로에 닿지 않는다.
  • Core/GameFlow 순수성 — 두 asmdef는 noEngineReferences:true. 셸·비주얼 어떤 것도 역참조를 만들지 않는다.
  • EditMode 테스트 경계 — ScriptedDecisionAgent + onApply=null로 비트동일 검증하는 헤드리스 경로. DI/씬/애니가 이 경로를 참조하면 안 된다.
  • TableViewModel 계약 — IHumanInputPort 6메서드, ApplySnapshot의 손패 사본·CurrentTrick null→값 강제통지(R3 변경감지 필수), 턴 결정 캐시, 폭탄 인터럽트 linked CTS. 무수정.

어셈블리 / 레이어 — 무엇이 추가되나

무변경(불변식) 무변경(불변식) 기존 asmdef · 폴더만 추가 신규(셸/패키지/순수 reducer)

의존은 안쪽(Core)으로만 흐른다. 새 asmdef는 0개 — 셸·비주얼·오디오·스텁 서비스·순수 reducer 모두 기존 Tichu.Presentation 안에 폴더로 분리. 패키지 2종(VContainer·DoTween)만 추가.

왜 새 asmdef 0개인가(②안 기각 핵심). ②안의 Tichu.App+Tichu.Services+Tichu.Network 3개 신설은 Phase 1 실범위(싱글+AI3 로컬)에 비해 명백한 Phase 2 선투자다(Network는 Phase 1 미사용 빈 골격). 새 asmdef는 컴파일 경계·순환참조·테스트 참조 갱신 비용만 늘린다. 격리가 정말 필요해지는 Phase 3(온라인)에서 폴더를 asmdef로 승격하면 비파괴적이다.

5 P1-C — 앱 셸 / 흐름 / DI

5.1 앱 흐름 — 순수 reducer + R3 상태 스트림

화면 전이를 순수 함수 AppFlowReducer.Reduce(state, event) → state로 모델링한다(③안 이식). 게임 규칙(이미 Core 소유)과 무관한 별개 FSM이라 UnityEngine 없이 EditMode에서 단위 테스트한다. 부수효과(씬 로드·패널 토글·전환 트윈·BGM)는 AppFlowMachine이 ReactiveProperty<AppFlowState> 스트림을 구독해 수행 — 기획서의 IReadOnlyReactiveProperty<ScreenState> 요구에 정확히 부합.

AI전만 활성 경로. 일시정지는 별 상태가 아닌 IsPaused 직교 플래그로 두어 게임 루프와 분리.

5.2 씬/패널 — 하이브리드(메뉴=패널, 인게임=씬)

씬은 단 1개만 추가한다. 메뉴/결과/일시정지는 영속 Canvas 위 패널 프리팹(SetActive 토글)로 — 씬 로드 비용 0, 전환 트윈 자유. 무거운 인게임 테이블만 Table.unity를 Additive 로드/언로드해 메모리 회수를 명확히 한다. 세 모드는 모드별 씬을 만들지 않고 GameLaunchArgs DTO로 단일 Table 진입에 수렴(기획 "동일 4석 로비" 요구).

메뉴는 가볍게(패널), 인게임은 명확하게(씬). 모드별 씬 폭발 없이 GameLaunchArgs로 수렴.

5.3 DI — VContainer 1스코프, 단명 객체는 직접 new 유지

VContainer를 "타입 의존이 아닌 와이어링 편의"로만 채택한다(③안 이식). 앱-수명 서비스만 AppScope에 등록하고, AsyncGameDriver/TableViewModel/에이전트는 라운드마다 시드로 new되는 단명 객체라 컨테이너에 올리지 않는다(현 RoundBootstrap의 직접 new 유지). 모든 타입은 POCO+생성자 주입이라 컨테이너 없이도 그대로 new 가능 → 기존 테스트 무파괴.

DI는 앱-수명 서비스에만. 단명 게임 객체를 DI에 올리면 수명관리 복잡도만 커진다(②안이 떠안은 부담을 회피).

패키지 도입 시 R3 참조 점검(검증 반영). Tichu.Presentation.asmdef는 R3를 명시 참조하지 않고 autoReferenced 경유로 쓰고 있다(references = Core·GameFlow·UniTask·Unity.InputSystem). VContainer 추가 시 컴파일 안정성을 위해 R3.Unity 명시 참조도 함께 점검할 것. ✓ 결정(2026-06-15): VContainer 채택 확정 — git URL로 manifest.json에 추가(R3·UniTask와 동일 방식). plain CompositionRoot 폴백은 미채택.

6 P1-D — 비주얼 / 애니 / 사운드 / 모바일

6.1 애니메이션 — "진실/연출 분리" (가장 중요한 결정)

핵심 난제: 현 onApply은 스냅샷 즉시 전체 갱신인데, 트윈은 시간을 소비하는 점진 변화다. 정석은 두 책임 분리 — (A) 논리 진실은 ApplySnapshot이 즉시 갱신(R3가 진실의 원천), (B) 시각 연출은 이미 존재하는 vm.Played(액션 스트림)를 ICardAnimator가 구독해 트윈 큐로 재생한다. 절대 금지: onApply에서 트윈 완료를 await하는 것(드라이버를 가로채 오라클·결정 순서 오염).

onApply=null이면 RecordPlay가 호출되지 않아 Played가 발화하지 않는다 → 애니/오디오가 오라클 경로에 닿지 않음(코드로 확인: AsyncGameDriver.cs:173, RoundBootstrap.cs:77).

6.2 60fps — 카드 풀링 + Canvas 분할

현 코드의 실측 핫스팟은 RenderHand/Trick/Backs가 매 스냅샷마다 자식을 Destroy + 재생성하는 것(GC 스파이크·레이아웃 리빌드). 프리팹화하면서 카드 칩 오브젝트 풀(Destroy 대신 SetActive 토글 + 위치만 갱신)로 전환하고, 정적(점수판/배경) vs 동적(손패/트릭) Canvas를 분리해 손패 갱신이 전체 캔버스를 dirty시키지 않게 한다.

추측이 아니라 측정된 핫스팟 대응. 풀 반환 시 onClick.RemoveAllListeners 누락은 흔한 함정 — 체크리스트로 가드.

6.3 가로 전환 · SafeArea

  • 방향 고정: PlayerSettings에서 세로/세로플립 비허용, LandscapeLeft/Right만 허용. Application.targetFrameRate=60, vSync off.
  • CanvasScaler 재설계: 현 1080×1920(세로)·match 0.5 → 1920×1080(가로)·matchWidthOrHeight=1(높이 기준). 이 전환은 TableUiView가 절대좌표를 코드로 계산하므로 반드시 프리팹화(앵커 기반)와 동시에 진행해야 화면이 깨지지 않는다(§8 충돌 C-1 참조).
  • SafeArea: SafeAreaFitter 컴포넌트(Screen.safeArea→RectTransform 보정)로 노치/펀치홀 회피(②안 이식).

6.4 프리팹 마이그레이션 — ITableView 2단계 전환

TableUiView(런타임 빌드, 약 770줄)를 한 번에 갈아엎지 않는다(③안 이식). 1단계: ITableView 인터페이스를 추출하고 현행 구현을 RuntimeTableView로 둔다 — vm 구독 계약을 인터페이스로 못박는다. 2단계: PrefabTableView(SerializeField 자식·실제 스프라이트·풀링·가로 앵커)로 교체. TableViewModel은 한 줄도 바뀌지 않는다.

구독 계약 체크리스트(②안 이식, 누락 시 UI desync). 새 뷰는 다음 11종을 빠짐없이 구독해야 한다 — Phase · CurrentTurn · MyHand · CurrentTrick · RoundResult · PendingDecision · Wish · CumulativeA/B · HandCount[4] · TichuAvailable · Played · PlaysCleared · BombReserved · FastForward. 인터페이스로 컴파일 강제할 것.

6.5 사운드

IAudioService(PlaySfx/PlayBgm/볼륨) + AudioBank(ScriptableObject, SfxId→Clip) + AudioSource 풀(SFX 라운드로빈 + BGM 1채널 크로스페이드). 훅은 애니와 동일 소스 — vm.Played/RoundResult/버튼 onClick 구독. AudioBank 미할당이면 no-op(클립 null 가드)이라 사운드 에셋 없이도 P1-C 진행 가능. 테스트는 NoOpAudioService 주입.
