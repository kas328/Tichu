난이도 선택 UI — 설계 제안

2026-07-04 · 승인 전 설계 · 앱 셸(VContainer DI + AppFlowMachine R3, 코드생성 메뉴)
요지. 현재 GameSessionPresenter가 new GameLaunchArgs()(=Normal 고정)로 매치를 시작한다. 셸 주석이 "AI 난이도는 순수 reducer가 아니라 셸이 GameLaunchArgs로 별도 운반"이라 명시 → 난이도를 이벤트가 아닌 셸 상태 홀더(MatchSettings)로 들고 가는 게 의도된 설계. 남은 결정은 선택 UI를 흐름 어디에 넣을지다.

1. 현재 흐름 & 삽입 지점

2. 배치 3안

A. 전용 "난이도 선택" 화면 추천

새 ScreenState.DifficultySelect 패널에 4버튼(쉬움/보통/어려움/전문가) + 뒤로. "AI 대전" → 난이도 화면 → 탭 시 그 난이도로 시작.

  • ✔ 기존 패널/AddButton 인프라 그대로 재사용(신규 UI 코드 최소). 순수 reducer엔 전이 3개만 추가.
  • ✔ 발견성 최고·표준 UX. 난이도 선택이 눈에 확실히 보임.
  • △ 시작까지 탭 1회 증가(게임시작→모드→난이도→게임). 난이도 선택엔 표준적이라 수용 가능.

B. 모드 선택에 난이도 버튼 인라인 대안

모드 선택 패널에서 "AI 대전"을 4버튼("AI 대전 · 보통" 등)으로 펼치거나, 난이도 사이클러 컨트롤 + 시작 버튼.

  • ✔ 새 화면 없음·탭 수 적음.
  • ✗ 모드 선택 패널이 이미 랭킹/친구방/뒤로로 붐빔 → 난이도까지 섞이면 어수선.
  • ✗ 컴팩트 사이클러는 MenuShellView에 없는 새 컨트롤 타입 필요(=A보다 코드 많음).

C. 설정(Settings)에 난이도 옵션 대안

설정 화면에 난이도 선택 컨트롤. "AI 대전"은 현재 설정값을 사용. PlayerPrefs로 앱 재시작에도 유지.

  • ✔ 선호가 기억됨·모드 흐름은 깔끔.
  • ✗ 설정 화면엔 지금 "뒤로"뿐 → 새 컨트롤 타입 필요.
  • ✗ 매치 시작 지점에서 덜 보임(플레이어가 못 찾을 수 있음)·지속화로 범위↑.

3. 난이도 운반 방식 (공통·확정)

어느 배치든 동일: 신규 앱-수명 홀더 MatchSettings { Difficulty = Normal }를 AppScope에 싱글톤 등록. 선택 UI가 _settings.Difficulty = 선택값 기록 → GameSessionPresenter가 new GameLaunchArgs { Difficulty = _settings.Difficulty }로 시작. 순수 AppFlowReducer/이벤트는 ScreenState만 다루고 난이도는 안 실음(주석 의도 준수).

지속성: MVP는 세션 인메모리(앱 실행 중 마지막 선택 기억, 기본 Normal). 앱 재시작 유지(PlayerPrefs)는 쉬운 후속 옵션 — 기본은 YAGNI로 미포함.
