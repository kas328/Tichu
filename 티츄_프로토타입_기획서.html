1 프로젝트 개요

정식 룰 기반 4인 카드게임 "티츄(Tichu)"를 모바일에서 누구나 쾌적하게 즐기도록 만든다. 4인 실시간 멀티플레이(랭킹전/친구방)와 고도화된 AI 상대(싱글)를 한 코어 위에서 제공하며, 4인 화면의 가독성, 정식 룰의 정확성, 강한 AI를 최우선 가치로 둔다.

이 문서는 프로토타입 단계의 검토용 상세 기획서다. 본 섹션은 비전·핵심 가치·기술 스택·모드 범위를 정의하고, 룰/턴 흐름은 3장, 화면·UX는 6장, 네트워킹·백엔드는 9장에서 상세화한다.

1.1 비전과 목표

  • 한 줄 비전: "모바일에서 가장 읽기 쉽고, 룰에 가장 정확하며, 가장 똑똑한 AI와 둘 수 있는 티츄."
  • 프로덕트 목표: 4인 실시간 동기 플레이 + 끊김 없는 싱글(vs AI)을 동일 게임 로직(룰 엔진)으로 구동해, 온라인/오프라인 경험 격차를 최소화한다.
  • 프로토타입 목표(이 문서 범위): 정식 룰 1판 완주(파트너십 점수 계산·티츄/그랜드티츄 선언·봄/용/개/마작 특수카드 포함)와 AI 4인 자동 대국, 가로 화면 4인 레이아웃 검증까지.

1.2 핵심 가치

가독성 최우선 P0

좁은 모바일 화면에 4인의 손패 수·콤보·점수·선언 상태를 한눈에. 카드 정렬, 콤보 하이라이트, 턴 인디케이터를 시각 우선순위로 설계한다. (상세 6장)

정식 룰 준수 P0

국제 티츄 정식 룰을 기준으로 콤보 판정·트릭 승계·특수카드·점수 정산을 엄격히 구현. 룰 엔진은 네트워크/AI와 분리해 단위 테스트로 검증. (상세 3장)

고도화된 AI P0

난이도별 정책(휴리스틱 → 확률/몬테카를로 기반)으로 카드 카운팅·파트너 협력·티츄 선언 판단을 수행. 싱글의 핵심 재미이자 멀티 대전 보조. (상세 7장)

깔끔한 디자인 P1

tichu.be의 정돈된 톤을 레퍼런스로 하되, 모바일 가로 4인에 맞춰 정보 밀도와 터치 타깃을 재설계. 일관된 컬러·타이포·모션 시스템 적용.

2 게임 규칙 ① 덱 구성과 특수 카드

티츄의 모든 로직은 56장 덱과 4종의 특수 카드 위에 세워집니다. 이 섹션은 덱 구성, 점수 카드의 분포(총 100점), 그리고 게임 흐름을 직접 좌우하는 특수 카드 4종(마작·개·봉황·용)의 정확한 효과를 정의합니다. 클라이언트의 카드 모델(CardModel), 점수 계산기(ScoreCalculator), AI의 가치 평가가 모두 이 정의를 단일 소스로 참조해야 합니다.

2.1 덱 구성 (56장)

일반 카드 52장

4개 문양 × 13개 랭크(2~10, J, Q, K, A)로 구성됩니다. 문양 간 우열은 없으며, 받기(받아치기)는 같은 타입·같은 장수의 더 높은 값으로만 가능합니다. 문양은 스트레이트 플러시(폭탄) 판정에서만 의미를 가집니다.

특수 카드 4장

마작(Mahjong, 값 1) · 개(Dog) · 봉황(Phoenix) · 용(Dragon). 일반 랭크 체계 바깥에 위치하며 각각 고유한 진행 규칙을 가집니다. 점수와 값은 아래 표/카드 컴포넌트에서 정확히 정의합니다.

값 1 = 마작 값 없음 = 개 와일드 = 봉황 최강 단독 = 용

2.3 특수 카드 4종 정밀 명세

마작 Mahjong 값 1 0점

  • 가장 낮은 값(1). 단독 최저이며 스트레이트의 시작(1~A)에 사용 가능.
  • 이 카드를 가진 플레이어가 첫 트릭의 선(리드)이 됩니다(한 판 시작 시 강제 첫 수).
  • 낼 때 소원(wish)으로 특정 숫자(2~A)를 지정할 수 있습니다.
  • 소원 지정 시: 이후 그 숫자를 합법적으로 낼 수 있는 플레이어는 반드시 그 숫자를 포함해 내야 합니다. 소원은 충족될 때까지 유지됩니다(다음 트릭으로 이월).

개 Dog 값 없음 0점

  • 선(리드)으로만, 단독으로만 낼 수 있습니다. 어떤 트릭에도 받아칠 수 없습니다.
  • 즉시 선을 파트너에게 넘깁니다. 파트너가 이미 아웃이면 파트너 다음 사람(파트너 기준 시계방향 다음)에게 넘어갑니다.
  • 트릭을 형성하지 않습니다 — 누구도 받지 않고, 테이블에 점수도 쌓이지 않으며, 파트너가 새 선으로 시작합니다.

봉황 Phoenix 와일드 -25점

  • 만능(와일드) — 폭탄을 제외한 어떤 조합에서도 임의의 일반 카드를 대체합니다(페어·트리플·풀하우스·스트레이트·연속 페어의 빈 자리 1장).
  • 단독으로 낼 때: 직전 단독보다 0.5 높은 값(예: 직전 K면 13.5). 단, 용은 이길 수 없습니다.
  • 선으로 처음 낼 때(직전 단독 없음): 값은 1.5(마작 1보다 높고 2보다 낮음).
  • 폭탄에는 사용 불가(포카드·스트레이트 플러시에 끼워 넣을 수 없음).

용 Dragon 최강 단독 +25점

  • 가장 높은 단독 — 에이스보다 높고 봉황을 포함한 모든 단독을 이깁니다.
  • 단독으로만 낼 수 있습니다(어떤 조합에도 포함 불가).
  • 용으로 트릭을 이기면 그 트릭 전체를 상대팀의 한 명에게 양도해야 합니다(자기 팀 점수로 가져갈 수 없음).
  • 다음 선은 정상적으로 용을 낸(=트릭을 이긴) 플레이어가 가져갑니다.

구현상 명확한 점

  • 점수는 카드 메타로 100점 고정 — 정산 검증 어서션 용이.
  • 마작/개의 트리거(첫 선/선 이양)는 결정론적이라 테스트 케이스화 쉬움.
  • 용 단독 최강·봉황 와일드는 비교 함수(CompareTrick)에 특수 분기로 격리 가능.

엣지 케이스 주의

  • 봉황 단독값은 정수가 아닌 0.5 가산이라 값 타입을 float 또는 (정수×2) 인코딩으로 설계해야 함.
  • 개의 선 이양 시 파트너 아웃 분기(다음 사람)를 빠뜨리기 쉬움.
  • 소원(wish)은 트릭 간 이월 + "합법적으로 낼 수 있을 때만 강제" 조건이 까다로움.
  • 용으로 이긴 트릭의 점수는 양도 후 상대팀에 귀속 — 점수 집계 순서 주의.
봉황은 와일드지만 폭탄(포카드·스트레이트 플러시)에는 절대 포함될 수 없습니다. 또한 봉황 단독은 용을 이기지 못하며, 선으로 처음 낼 때만 값 1.5로 취급합니다. AI의 봉황 가치 평가에서 이 두 제약을 반드시 반영하세요.
다음 섹션(3. 조합과 폭탄, 4. 한 판 진행/정산)에서 본 절의 값 정의를 받아 받기 규칙·폭탄 우열·티츄 콜 정산을 다룹니다. 봉황의 와일드 대체와 마작의 스트레이트 시작은 조합 판정 로직에서 재참조됩니다.

3 게임 규칙 ② 카드 조합과 폭탄

티츄의 트릭은 "선(lead)이 정한 조합 타입과 장수"를 따라가며, 같은 타입·같은 장수의 더 높은 값으로만 받을 수 있습니다. 이 섹션은 6종 기본 조합과 봉황 와일드, 그리고 트릭 흐름을 무시하고 끼어드는 폭탄 2종의 정의·우선순위·검증 규칙을 코드 착수가 가능한 수준으로 정리합니다. 조합 판정(ComboValidator)과 받기 가능 여부(BeatChecker)는 클라이언트 입력 가드와 서버 권위 검증 양쪽에서 동일 로직을 공유해야 합니다.

3.3 폭탄(Bomb) 2종과 우선순위

폭탄은 일반 조합 흐름을 무시하는 특수 조합으로, 두 종류가 있고 명확한 강도 순서를 가집니다. 봉황은 어떤 폭탄에도 포함될 수 없습니다.

Bomb · 약 포카드(Four of a Kind)

  • 같은 랭크 4장(문양 4종 모두).
  • 폭탄이 아닌 모든 조합을 값과 무관하게 이김.
  • 포카드끼리는 랭크가 높은 쪽이 이김.

Bomb · 강 스트레이트 플러시(Straight Flush)

  • 같은 문양 연속 5장 이상.
  • 포카드보다 강함 — 포카드를 항상 이김.
  • 스트레이트 플러시끼리는 더 긴 것이 우선, 같은 길이면 더 높은 값이 우선.

폭탄 우선순위(받기 비교 알고리즘)

  1. 새 조합이 폭탄이 아니고 현재 트릭 마지막도 폭탄이 아니면 → 같은 타입·장수의 더 높은 값으로만 받기.
  2. 새 조합이 폭탄이고 마지막이 비폭탄이면 → 항상 받기 성공(인터럽트 가능).
  3. 둘 다 포카드면 → 랭크 비교.
  4. 마지막이 포카드, 새 조합이 스트레이트 플러시면 → 받기 성공.
  5. 마지막이 스트레이트 플러시, 새 조합이 포카드면 → 실패.
  6. 둘 다 스트레이트 플러시면 → (길이 우선, 동일 길이 시 최고 랭크) 비교.
// BeatChecker.cs (의사 코드)
public static bool CanBeat(Combo prev, Combo next)
{
    // 폭탄 인터럽트: 차례 무관하게 호출됨 (TurnGuard 우회)
    bool prevBomb = prev.IsBomb, nextBomb = next.IsBomb;

    if (!nextBomb)
    {
        if (prevBomb) return false;                 // 비폭탄은 폭탄 못 받음
        if (prev.Type != next.Type) return false;   // 타입 일치
        if (prev.Length != next.Length) return false; // 장수 일치
        return next.Value > prev.Value;             // 더 높은 값만
    }
    // next는 폭탄
    if (!prevBomb) return true;                      // 비폭탄 → 폭탄: 무조건 성공
    // 폭탄 vs 폭탄: 강도(Tier)로 1차 비교
    if (next.BombTier != prev.BombTier)
        return next.BombTier > prev.BombTier;        // StraightFlush(2) > FourKind(1)
    // 같은 Tier
    if (next.BombTier == BombTier.StraightFlush)
        return next.Length != prev.Length
             ? next.Length > prev.Length             // 더 긴 것 우선
             : next.Value > prev.Value;              // 동일 길이면 높은 값
    return next.Value > prev.Value;                  // 포카드끼리는 랭크
}

폭탄 인터럽트(차례 외 투입)

폭탄은 자기 차례가 아니어도 즉시 끼어들어 낼 수 있습니다. 이는 일반 턴 순서(TurnGuard)를 우회하는 유일한 합법 액션이므로, 네트워크/UI 설계에서 별도 처리가 필요합니다.

인터럽트 처리 흐름(권위 서버 기준). 임의 시점에 폭탄 투입 입력이 들어오면 서버는 (1) 현재 트릭 마지막 조합 대비 CanBeat 통과 여부, (2) 손패 보유 여부를 검증하고, (3) 통과 시 진행 중이던 입력 윈도우를 취소하고 트릭의 새 리드를 폭탄 투입자로 갱신합니다. 동시에 여러 명이 폭탄을 던지는 경합은 서버 수신 순서로 직렬화하되, 더 강한 폭탄이 약한 폭탄을 다시 덮는 재인터럽트를 허용합니다. UI는 자기 턴이 아니어도 손에 합법 폭탄이 있으면 항상 "폭탄!" 버튼을 노출하고, UniRx로 트릭 상태를 구독해 실시간 활성/비활성을 토글합니다.

폭탄 강도 순서(포카드 < 스트레이트 플러시)와 차례 외 인터럽트 흐름.

3.4 받기 규칙 요약과 검증 체크리스트

받기(beat)는 트릭의 핵심 규칙입니다. 일반 조합은 선이 정한 타입을 따라가며, 폭탄만이 타입 경계를 넘습니다.

합법(받기 성공)

  • 같은 타입·같은 장수의 더 높은 값.
  • 비폭탄 위에 폭탄 투입(차례 무관).
  • 포카드 위에 더 높은 포카드 또는 임의의 스트레이트 플러시.
  • 스트레이트 플러시 위에 더 길거나(같은 길이면 더 높은) 스트레이트 플러시.

불법(받기 실패)

  • 다른 타입 또는 다른 장수(폭탄 제외).
  • 같거나 낮은 값.
  • 스트레이트 플러시 위에 포카드.
  • 봉황을 폭탄에 포함.
  • 봉황을 한 조합에 2장 이상(불가능하지만 가드 필요).
구현 합의. ComboValidator와 BeatChecker는 순수 함수로 작성해 클라이언트(입력 가드/하이라이트)와 서버(권위 검증) 양쪽에서 동일 어셈블리를 참조합니다. 단위 테스트는 "조합 인식 정확성"과 "받기 비교 정확성"을 분리해 작성하고, 봉황 대체·폭탄 인터럽트·계단/스트레이트 길이 비교를 골든 케이스로 고정합니다. 마작(소원) 강제·개·용 등 특수 카드의 트릭 흐름 규칙은 다음 섹션(트릭 진행)에서 다룹니다.

4 게임 규칙 ③ 진행과 점수 계산

한 판(Round)은 8장 딜 → 큰 티츄 콜 → 추가 6장 딜 → 카드 교환 → 마작 선 → 트릭 루프 → 아웃 판정 → 정산의 고정된 순서로 흐른다. 본 섹션은 각 단계의 트리거 조건과 점수 계산 규칙을 상태 머신 관점에서 정의하여, RoundController / ScoreCalculator 구현 시 그대로 매핑할 수 있도록 한다. 게임(Game)은 한 팀이 1000점에 도달할 때까지 여러 판을 반복한다(목표 점수 설정 가능).

4.1 한 판 진행 순서

  1. 8장 딜 — 각 플레이어에게 8장을 나눠준다. 이 8장만 본 상태에서 다음 단계의 큰 티츄 콜 여부를 결정한다.
  2. 큰 티츄(Grand Tichu) 콜 — 9번째 카드를 받기 전, 8장만 본 상태에서만 콜 가능(±200). 각 플레이어는 나머지 6장을 받기로 확정하는 순간 더 이상 큰 티츄를 콜할 수 없다.
  3. 나머지 6장 딜 — 총 14장이 된다.
  4. 카드 교환(Exchange) — 각자 왼쪽 / 파트너 / 오른쪽 세 명에게 1장씩 골라 전달한 뒤 동시에 공개·수령. 보낸 3장과 받은 3장이 1:1로 맞교환되므로 교환 후에도 손패는 14장.
  5. 마작 선(Lead) — 마작(값 1)을 가진 플레이어가 첫 트릭의 선이 된다.
  6. 작은 티츄(Tichu) 콜 — 자신의 첫 카드를 내기 전이라면 언제든 콜 가능(±100). 이미 한 장이라도 낸 플레이어는 콜 불가.
  7. 트릭 루프(Trick Loop) — 선이 합법 조합을 내면, 시계 방향으로 같은 타입·같은 장수의 더 높은 값 또는 폭탄으로 받거나 패스. 전원 패스 시 마지막으로 낸 사람이 트릭을 가져가고 다음 트릭의 선이 된다. (개·용·봉황·소원 등 특수 처리는 §3 참조)
  8. 아웃(Out) — 손패를 모두 낸 순서대로 1·2·3등이 확정된다.
  9. 종료 판정 — 3명이 아웃되면 종료. 단, 한 팀의 두 명이 1·2등으로 먼저 빠지면(원-투, One-Two) 즉시 종료.
  10. 정산(Scoring) — 아래 4.4 규칙으로 양 팀 점수를 합산하고, 큰/작은 티츄 성패를 가산한다.

(a) 한 판 플로우: 8장 딜 → 큰 티츄 콜 → 추가 6장 딜 → 교환 → 마작 선 → 트릭 루프(전원 패스 시 다음 선으로 순환) → 아웃/종료 판정 → 정산 → 1000점 체크.

4.2 큰 티츄 / 작은 티츄 콜

티츄 콜은 "이번 판에서 내가 가장 먼저(1등으로) 아웃하겠다"는 베팅이다. 성공/실패는 카드점수와 독립적으로 가산된다. 콜 타이밍 검증은 서버 권위(Authoritative)로 처리하며, 클라이언트는 UI 게이팅만 담당한다.

구분콜 가능 시점성공 조건성공실패
큰 티츄 (Grand) 8장만 본 상태(9번째 카드 수령 전) 해당 플레이어가 그 판 1등으로 아웃 +200 −200
작은 티츄 (Tichu) 자신의 첫 카드를 내기 전이면 언제든 해당 플레이어가 그 판 1등으로 아웃 +100 −100
큰 티츄와 작은 티츄는 중복 적용되지 않는다. 큰 티츄를 콜한 시점에 이미 ±200 베팅이 확정되므로, 같은 판에서 작은 티츄를 추가로 콜할 수 없다(UI에서 큰 티츄 콜 시 작은 티츄 버튼 비활성화).
구현 함정 — "1등 아웃"의 정의: 성공은 그 판에서 가장 먼저 손패를 비운 플레이어일 때만이다. 파트너가 먼저 아웃하면 콜한 본인은 자동 실패다. 또한 원-투로 판이 즉시 끝나도, 티츄 성패는 정산 시 별도로 가산된다(원-투 +200과 합산 가능).

4.3 카드점수와 특수 카드 점수

카드점수 (덱 총합 100점)

  • 모든 K = 10점 (4장)
  • 모든 10 = 10점 (4장)
  • 모든 5 = 5점 (4장)
  • 용(Dragon) = +25점
  • 봉황(Phoenix) = −25점

합계 = (10·4)+(10·4)+(5·4)+25−25 = 100점. 즉 한 판에서 카드점수만으로 양 팀이 나눠 갖는 총량은 항상 100이다(원-투 케이스 제외).

점수 무관 카드

  • 마작(1), 개(Dog): 0점
  • 2~9, J, Q, A: 0점
봉황(−25)은 "딴 트릭에 들어 있으면 그 팀 점수를 깎는다"는 의미다. 트릭 점수 합산 시 단순 가산하면 자동으로 처리되므로 별도 분기 불필요.

4.4 정산 규칙

판이 종료되면 정산은 다음 두 갈래로 나뉜다.

CASE A 원-투 (One-Two)

한 팀의 두 명이 1·2등으로 먼저 아웃 → 판 즉시 종료.

  • 해당 팀: +200 고정
  • 상대 팀: 0
  • 그 판의 카드점수는 전부 무시한다(트릭/잔여카드 합산 안 함).
  • 티츄/큰 티츄 성패는 별도로 가산.

CASE B 일반 종료

3명이 아웃되어 마지막 1명이 남은 경우, 그 플레이어 기준으로 분배한다.

  1. 잔여 손패 점수 → 상대 팀에게 전달.
  2. 자신이 딴 트릭 → 1등으로 아웃한 플레이어에게 전부 넘김.
  3. 각 팀이 그 판에서 확보한 트릭의 카드점수를 합산.

그 다음 양 팀의 카드점수에 티츄/큰 티츄 성패(±100/±200)를 더한다.

(b) 정산 분배: 마지막 플레이어의 ① 잔여 손패 카드점수는 상대 팀으로, ② 자신이 딴 트릭은 1등 아웃 플레이어(=그 팀)로 흘러간다. 이후 팀별로 카드점수를 합산하고 티츄 성패를 더한다.

4.5 게임 종료

장점

  • 1000점 누적 종료는 역전 가능성을 남겨 긴장감을 유지한다.
  • 목표 점수를 GameConfig.targetScore로 노출하면 빠른 매치(500점) 등 모드 확장이 쉽다.

단점 / 주의

  • 한 판에서 양 팀이 동시에 1000을 넘으면 최고 점수 팀 승리로 타이브레이크 규칙을 명시해야 한다.
  • 온라인전에서 판 수가 가변이므로, 세션 시간 예측이 어렵다 → 평균 판 수 텔레메트리로 매칭 UX 보정 필요.
네트워크 권위 원칙: 콜 타이밍·합법 수·정산은 모두 서버에서 계산하고, 클라이언트는 결과를 수신해 DoTween 애니메이션으로 표현만 한다. 클라이언트 점수 계산을 신뢰하면 랭킹전 어뷰징에 직결되므로, ScoreCalculator는 서버/클라 공유 코드로 두되 최종 권위는 서버 인스턴스로 한정한다.

5 화면 구조 & 흐름

앱은 인트로 → 메인(허브) → 모드 진입(랭킹전 / AI전 / 친구와 게임) → 인게임 테이블의 단일 진입 트리로 구성된다. 모바일 우선 설계로, 메인 허브에서 한 번의 탭으로 핵심 동선(게임 시작·랭킹·게임 방법)에 도달하는 것을 목표로 한다. 화면 전이는 UniRx의 상태 스트림(IReadOnlyReactiveProperty<ScreenState>)으로 관리하고, 전환 연출은 DoTween, 매치메이킹·방 참가 등 네트워크 대기 구간은 UniTask로 처리한다.

본 섹션은 화면 계층과 전이 규칙에 집중한다. 각 화면의 레이아웃 상세(4인 테이블 가독성, 핸드/플레이 영역)는 별도 인게임 섹션에서 다루며, BM 관련 화면(상점·광고)은 본 프로토타입에서 구조만 정의하고 기능은 보류한다.

5.1 사이트맵 (화면 전이)

6 인게임 UI/UX

인게임 화면의 제1원칙은 가독성이다. 4명이 동시에 등장하는 카드게임에서 플레이어는 "지금 누구 차례인지, 무엇을 받아야 하는지, 내 손에서 무엇을 낼 수 있는지"를 한눈에 파악해야 한다. 본 섹션은 화면 방향·좌석 배치·트릭 영역·콜/특수 카드 UI·카드 애니메이션(DoTween) 가이드를 정의하고, tichu.be의 정돈된 톤을 레퍼런스로 하되 모바일 가로에 맞춰 개선한다.

결정: 인게임은 가로(Landscape) 고정 권장. 4인을 상/하/좌/우 네 좌석으로 배치하려면 가로 화면이 압도적으로 유리하다(세로에서는 좌·우 상대와 중앙 트릭을 동시에 보기 어렵다). 손패 부채꼴의 가시 폭, 카드 터치 타깃, 콤보 하이라이트 모두 가로에서 확보된다. 로비·메인·상점 등 비전투 화면은 세로 병행을 검토하되, 테이블 진입 시 가로로 전환한다. → 9·13장 승인 항목

6.1 4인 좌석 배치와 화면 구성

본인은 항상 하단 중앙(손패 부채꼴), 파트너는 상단, 두 상대는 좌·우에 고정 배치한다. 절대 좌석이 아니라 "나 기준 상대 좌석"으로 회전시켜, 어떤 플레이어든 자기 시점에서 동일한 멘탈 모델을 갖게 한다(파트너=정면).

(a) 인게임 가로 와이어프레임 — 본인(하단)·파트너(상단)·상대(좌/우) 고정 배치, 중앙 트릭 영역, 손패 부채꼴(선택 카드 돌출), 우하단 액션 버튼(티츄/내기/패스/폭탄).

6.2 가독성 설계 원칙

시선 우선순위

① 내 차례 여부 ② 내가 낼 수 있는 카드(콤보 하이라이트) ③ 트릭 마지막 제출 ④ 상대 잔여 장수·티츄 콜 순으로 시각 가중치를 둔다.

손패 가시성

내 손패는 항상 가장 크고 선명하게. 자동 정렬(랭크/콤보별 토글), 선택 시 카드 돌출, 유효하지 않은 카드는 디밍 처리.

상대 정보 최소화

상대는 뒷면 스택 + 남은 장수 + 콜 뱃지만. 정보 과잉을 막아 중앙 트릭과 내 손패에 집중시킨다.

색·대비

문양색(옥/검/탑/별)으로 즉시 구분, 점수 카드(K·10·5·용·봉황)는 보조 뱃지로 강조. 명도 대비 WCAG AA 목표.

터치 타깃

카드 최소 터치 폭 확보, 부채꼴 겹침 구간은 히트박스를 위쪽 노출 영역으로 한정해 오터치 방지.

피드백 즉시성

탭·제출·획득·콜에 즉각적 모션/햅틱/사운드. 네트워크 지연 시 낙관적 UI + 서버 확정 후 보정.

(b) 탭으로 선택 → 콤보 판정/하이라이트 → 받기 가능 시 '내기' 활성 → 제출(중앙 이동) → 트릭·턴 갱신. 클라이언트는 낙관적으로 표현하고 최종 권위는 서버.

6.6 tichu.be 참고점과 개선 아이디어

tichu.be에서 가져올 점

  • 군더더기 없는 정돈된 레이아웃과 명확한 트릭 영역.
  • 콤보/콜 상태를 텍스트가 아닌 위치·뱃지로 전달.
  • 빠른 진행감(불필요한 연출 최소화).

모바일을 위한 개선

  • 데스크톱 클릭 → 터치 기준 히트박스·부채꼴 재설계.
  • 자동 콤보 추천/정렬 토글로 입력 부담 감소.
  • "낼 수 있는 카드" 자동 디밍 + 원탭 콜로 초보 접근성↑.
  • 딱딱하지 않은 톤: 부드러운 모션·캐릭터 이모트로 분위기 환기.
UI 상태는 UniRx 단방향 바인딩으로 구성한다: GameState(서버 동기) → ReactiveProperty → View. 손패·점수·턴·타이머·콜 뱃지는 상태 구독으로만 갱신해, 표현 로직과 게임 로직을 분리한다(상세 10장).

7 게임 진행 시퀀스 (시스템 관점)

티츄는 입력 빈도가 낮은 저틱레이트(low tick-rate) 턴제 카드게임이라 물리 동기화가 필요 없고 메시지가 경량입니다. 다만 랭킹전은 상대 손패를 가린 히든 핸드(hidden hand)와 치팅 방지가 핵심이므로, 서버가 전체 게임 상태를 보유하고 각 클라이언트에는 자기 시야만 전송하는 서버 권위(server-authoritative) 구조를 전제로 설계합니다. 본 섹션은 한 트릭(trick)·한 라운드(round)의 메시지 흐름, 상태 동기화 원칙, 예외 처리를 시스템 관점에서 정의합니다.

이 문서는 전송 기술(WebSocket/Photon/Nakama/UGS 등) 선택과 무관하게 성립하는 논리 프로토콜을 기술합니다. 구체 백엔드 선정은 8장(네트워킹 아키텍처)에서 다룹니다. 메시지 이름(PlayCards 등)은 식별자 예시이며 구현 시 확정합니다.

7.1 핵심 설계 원칙

권위 서버 단일 진실

전체 손패·덱·트릭 히스토리·점수는 서버에만 존재합니다. 클라이언트는 입력을 "요청"할 뿐, 게임 규칙 판정은 전적으로 서버가 수행합니다.

시야 부분 관측 전송

각 클라이언트에는 자기 손패만 전체 공개하고, 상대는 장수(count)만 전송합니다. 손패 카드 데이터가 절대 상대 클라이언트로 흐르지 않습니다.

경량 이벤트 기반

틱 루프가 아니라 이벤트(콜/플레이/패스) 단위로만 메시지가 오갑니다. 대역폭과 배터리 부담이 작아 모바일에 유리합니다.

안티치트 원칙: 서버는 모든 입력에 대해 (1) 발신자가 현재 행동권자인지 (2) 제출 조합이 손패에 실재하는지 (3) 직전 트릭을 이기는 유효 조합인지 (4) 폭탄/소원 등 특수 규칙을 위반하지 않는지 검증합니다. 하나라도 실패하면 상태 변경 없이 ActionRejected를 반환합니다.

7.3 클라이언트-서버 시퀀스 (한 트릭)

7.4 특수 규칙의 메시지 처리

폭탄 인터럽트

폭탄은 자기 차례가 아니어도 제출 가능합니다. 서버는 동시 도착한 폭탄 요청들을 수신 순서로 직렬화하고, 더 높은 폭탄만 인정한 뒤 차례를 폭탄 제출자로 재설정합니다. 클라이언트는 짧은 "폭탄 인터럽트 윈도우"에서만 폭탄 버튼을 활성화하되, 최종 판정은 서버가 합니다.

소원 (Mahjong Wish)

마작(1) 보유자가 리드 시 PlayCards.wishRank로 소원을 겁니다. 이후 서버는 각 플레이어의 차례마다 "소원 랭크를 낼 수 있는데 안 냈는지"를 검증하여, 소원을 만족할 수 있으면 강제합니다. 소원 충족 전까지 상태에 wish를 유지·방송합니다.

용 분배 (Dragon)

용으로 트릭을 이기면 그 트릭을 상대 팀 둘 중 한 명에게 줘야 합니다. 서버는 AssignDragon을 요청하고, 대상이 상대 팀인지 검증한 뒤 트릭 점수를 해당 플레이어 획득 더미로 이전합니다. 타임아웃 시 임의의 상대(좌측 상대 등 규칙 고정)로 자동 배정합니다.

티츄 / 큰 티츄 콜

큰 티츄는 DealHand(8장) 직후, 일반 티츄는 자신의 첫 카드 제출 전에만 유효합니다. 서버는 콜 시점·중복 콜을 검증하고, 콜 성공/실패를 라운드 정산 시 점수에 반영합니다(성공 +100/+200, 실패 -100/-200).

7.6 라운드 상태머신

한 라운드는 딜 → 큰 티츄 → 교환 → 플레이 → 정산의 결정적(deterministic) 상태 전이를 가집니다. 게임은 누적 점수가 목표(통상 1000점)에 도달할 때까지 라운드를 반복합니다. 서버는 이 FSM을 단일 진실로 구동하고, 각 전이 시점에만 클라이언트 입력을 수용합니다.

(b) 라운드 상태머신. 플레이 상태는 트릭 단위로 자기 순환하며, 라운드 종료 시 정산 후 목표 점수 도달 여부로 다음 라운드 또는 게임 종료로 분기합니다.

7.7 예외 처리

턴 타임아웃

  1. 서버는 행동권 부여 시 카운트다운(예: 일반 20초, 콜/교환 30초)을 시작하고 StateUpdate에 deadline을 실어 보냅니다. 클라이언트 타이머는 표시용이며, 만료 판정의 권위는 서버에 있습니다.
  2. 만료 시 행동에 따라 자동 처리: 팔로우 차례면 자동 패스, 리드 차례면 오토플레이(서버 규칙: 최저 단일 카드, 단 소원 충족 가능 시 소원 랭크 우선)를 적용합니다.
  3. 교환 미제출 시 규칙 기반 자동 교환(예: 무작위 또는 최저 카드 분배), 용 분배 미지정 시 고정 규칙(좌측 상대)으로 자동 배정합니다.
  4. 연속 타임아웃이 임계치(예: 3회)를 넘으면 해당 좌석을 "이탈(AFK)"로 표시하고 라운드 종료까지 오토플레이로 대체합니다.

연결 끊김 / 재접속

  1. 소켓 단절을 감지하면 서버는 해당 좌석을 즉시 비우지 않고 유예 시간(grace period, 예: 60초) 동안 좌석을 유지하며 상태를 보존합니다. 이 사이 해당 좌석의 차례는 타임아웃 자동 처리로 진행됩니다.
  2. 클라이언트는 재접속 시 Reconnect(sessionToken, lastSeq)를 보내고, 서버는 토큰을 검증한 뒤 현재 시야의 전체 스냅샷(StateSnapshot)을 전송합니다. 부분 델타가 아니라 전체 상태를 줘 동기화 일관성을 보장합니다.
  3. 유예 시간 초과 시 랭킹전은 기권/AI 인계 정책에 따라 처리(점수·MMR 패널티), 친구방/싱글은 호스트 권한 위임 또는 AI 인계로 처리합니다.

권한 검증 (부정 입력 거부)

서버가 항상 검증

  • 발신자 == 현재 행동권자(폭탄 예외 명시)
  • 제출 카드가 발신자 손패에 실재하는가
  • 조합 타입·세기가 직전 트릭을 합법적으로 이기는가
  • 소원 강제·콜 시점·교환 규칙 위반 여부
  • actionSeq 중복/역행 여부(리플레이 방지)

거부 처리 원칙

  • 실패 시 상태 변경 없이 ActionRejected(reason)만 발신자에게 반환
  • 거부는 브로드캐스트하지 않음(정보 노출 방지)
  • 비정상 빈도의 무효 요청은 로깅·레이트리밋·세션 제재
  • 클라이언트 UI는 사전 검증으로 막되, 이는 UX용일 뿐 신뢰 대상 아님
설계 요약: 클라이언트는 "입력 수집 + 결과 렌더링"만 담당하고, 규칙 판정·시야 마스킹·타이머·재접속 권위는 모두 서버에 둡니다. 이 경계가 분명할수록 히든 핸드 무결성과 랭킹전 공정성이 보장되며, 추후 8장의 어떤 백엔드(UGS/Nakama/커스텀 WebSocket)로도 동일 프로토콜을 이식할 수 있습니다.

8 AI 설계 (고도화)

티츄는 불완전 정보(상대 손패를 모름)와 파트너 협력이 동시에 작용하는 게임이다. 따라서 단순 룰 기반 AI로는 한계가 분명하며, 본 프로젝트는 도메인 휴리스틱과 결정화 몬테카를로 / ISMCTS(Information Set Monte Carlo Tree Search)를 결합한 표준 강 AI 접근을 채택한다. 모든 AI는 네트워크 플레이어와 동일한 의사결정 인터페이스를 통해 동일 코어 룰 엔진을 공유하여, 룰 정확성과 테스트 용이성을 보장한다.

핵심 설계 원칙: (1) AI는 별도 룰 구현을 갖지 않는다 — 사람과 동일한 IDecisionAgent로 게임에 참여한다. (2) 사고(thinking)는 메인 스레드를 절대 막지 않는다 — UniTask 백그라운드에서 시간 예산 내에 종료한다. (3) 강함은 "더 깊은 탐색"이 아니라 "더 나은 추론 + 협력"에서 나온다.

8.1 의사결정 파이프라인

AI는 턴이 돌아올 때마다 아래 6단계 파이프라인을 순차 실행한다. 앞 단계의 산출물(특히 확률 추론)이 뒤 단계의 입력이 되며, 평가 단계에서 ISMCTS 시뮬레이션을 통해 후보 수의 기대 가치를 산정한다.

그림 8-a. 카드 추적부터 콜 판단까지 6단계 파이프라인. 평가 단계가 ISMCTS의 핵심이며, 파트너 협력 보정이 팀 단위 기대가치를 산출한다.

1) 카드 추적/기억

  • 56장 전체 비트마스크(ulong)로 "이미 나온 카드" 관리
  • 각 플레이어가 어떤 트릭에 무엇을 냈는지 이력 보관
  • 특수 카드(개·용·봉황·1) 소진 여부 별도 플래그
  • "못 받고 패스" 정보로 보유 추정에 음의 단서 반영

2) 상대 손패 확률 추론

  • 미지 카드 풀을 상대 3인에게 분배하는 확률 분포 추정
  • 패스·콜·교환 단서로 베이지안 갱신(예: 빅티츄 콜 → 강패 보유 가능성↑)
  • ISMCTS의 결정화(determinization) 샘플링 사전분포로 사용

3) 합법 수 생성

  • 현재 트릭 상태에서 둘 수 있는 모든 조합 enumerate
  • 싱글·페어·트리플·풀하우스·스트리트·연속페어·폭탄 분류
  • 봉황 랭크 지정, 개/용 처리 등 특수 룰 포함
  • 패스도 항상 합법 후보로 포함(시작 트릭 제외)

4) 평가 (휴리스틱 + ISMCTS)

  • 1차: 빠른 휴리스틱으로 후보 가지치기(상위 K개만 탐색)
  • 2차: 결정화 표본마다 게임 트리 탐색, 시뮬레이션 통계 누적
  • 리프 평가에 손패 강도·콤보 구조·잔여 카드 가치 반영

5) 파트너 협력 보정

  • 보상을 개인 점수가 아닌 팀 점수로 정의
  • 파트너가 트릭을 가져갈 때 점수 카드를 "흘려주는" 가치 추가
  • 파트너 티츄 성공을 돕는 행동에 가중치 부여

6) 콜 판단

  • 교환 전/후 손패로 단독 1등 확률 추정 → 임계치 이상이면 티츄
  • 큰 티츄(Grand Tichu)는 8장 시점 더 보수적 임계치 적용
  • 폭탄 보유·선(lead) 확보 여부를 승률 보정 인자로 사용

8.2 파트너 협력 전술

티츄에서 팀워크는 단독 플레이보다 승패 기여도가 크다. AI는 평가함수가 팀 기대가치를 산출하므로 아래 전술이 명시적 규칙이 아니라 가치 보정으로 자연스럽게 발현되도록 설계한다(필요 시 강제 휴리스틱으로 보강).

점수 몰아주기 (Dumping)

파트너가 트릭을 확정적으로 가져갈 것으로 추정될 때, K/10/5 등 점수 카드를 자신이 따로 못 챙길 바엔 파트너 트릭 쪽으로 흘려보낸다. 용(25점)을 상대가 못 받을 상황에서 점수 손실 최소화도 동일 논리.

티츄 콜 보조 / 방해

파트너가 티츄를 콜하면 자신은 1등을 양보하고 파트너의 빠른 비움을 돕는다. 반대로 상대가 콜하면 선을 끊고 강패·폭탄으로 상대의 1등 달성을 방해한다.

선(Lead) 양보 / 확보

파트너가 손패를 비우는 데 유리하면 작은 콤보로 선을 넘겨준다. 반대로 자신이 빅티츄 중이면 선을 끝까지 쥐고 강제로 트릭을 정리한다.

폭탄 타이밍

폭탄은 트릭 순서와 무관하게 투입 가능하므로, "상대 티츄/큰 티츄 저지", "큰 점수 트릭 탈취", "파트너 위기 구원" 시점에만 가치가 임계치를 넘도록 평가. 무의미한 조기 소모를 억제한다.

협력 전술은 "정보 누설"과 충돌할 수 있다. 점수 몰아주기/선 양보가 상대에게 손패를 추론당하는 단서가 되지 않도록, Hard 이상에서는 협력 행동에 약간의 노이즈와 위장(bluff) 정책을 섞는다.

8.3 난이도 티어

4개 티어로 기법·강도·연산 예산을 차등한다. 하위 티어는 휴리스틱 위주로 빠르고 약하게, 상위 티어는 ISMCTS와 협력·블러프까지 활성화한다. 모든 티어는 동일 룰 엔진/인터페이스를 쓰며 차이는 정책(policy)과 예산뿐이다.

8.4 모바일 성능 전략

ISMCTS는 본질적으로 연산 집약적이므로, 모바일에서 프레임 히치 없이 동작시키기 위한 예산화·비동기·적응형 설계가 필수다.

시간 예산 제한 (Anytime)

티어별 상한 시간(예: Hard 250ms, Expert 500ms)을 두고, 시간이 소진되면 현재까지 최선 후보를 즉시 반환하는 anytime 알고리즘으로 구현. 시뮬레이션 수가 아닌 벽시계 시간을 1차 종료 조건으로 사용한다.

백그라운드 스레드 (UniTask)

탐색은 UniTask.RunOnThreadPool로 워커 스레드에서 실행하고, 결과만 메인 스레드로 마샬링한다. Unity 오브젝트에 접근하지 않는 순수 도메인 데이터(struct/배열)만 탐색에 사용해 스레드 안전을 확보한다.

적응형 시뮬레이션 수

기기 성능(프레임 타임 측정)과 국면 복잡도(합법 수 개수)에 따라 결정화 표본·플레이아웃 수를 동적 조절. 저사양 기기에서는 자동 다운스케일, 명백한 국면은 조기 종료한다.

"생각하는 척" 연출과 결합

UX상 즉답은 부자연스러우므로 최소 사고 시간(예: 0.4~0.8s)을 두되, 그 시간을 실제 탐색에 사용한다. DoTween 연출과 겹쳐 체감 지연을 숨긴다.

// AI 사고: 메인 스레드를 막지 않고 시간 예산 내에서 anytime 탐색
public async UniTask<PlayerAction> DecideAsync(GameStateView view, AiBudget budget, CancellationToken ct)
{
    // 1) 순수 데이터 스냅샷 (Unity 오브젝트 미참조 → 스레드 안전)
    var snapshot = view.ToImmutableSnapshot();

    // 2) 백그라운드 스레드에서 ISMCTS 실행 (시간 상한 = budget.MaxMs)
    var action = await UniTask.RunOnThreadPool(
        () => _searcher.RunIsmcts(snapshot, budget, ct), cancellationToken: ct);

    // 3) 최소 사고 시간 보장(체감 자연스러움) — 남는 시간만 대기
    await UniTask.Delay(budget.MinThinkMs, cancellationToken: ct);
    return action; // 동일 IDecisionAgent 계약으로 룰 엔진에 제출
}

8.5 룰 엔진 공유 (AI = 플레이어 동일 인터페이스)

AI와 네트워크 플레이어, 로컬 플레이어 모두 동일한 IDecisionAgent 계약으로 게임에 참여한다. 게임 진행 루프는 "누가 사람이고 누가 AI인지" 알 필요가 없으며, 동일한 코어 룰 엔진이 합법성 검증·점수 계산을 단독으로 담당한다. 이 구조는 (1) 룰 버그를 한 곳으로 수렴시키고, (2) 헤드리스 자가대국(self-play)으로 대량 시뮬레이션·밸런싱·회귀 테스트를 가능하게 한다.

// 사람·네트워크·AI가 공유하는 단일 계약
public interface IDecisionAgent
{
    UniTask<PlayerAction> DecideAsync(GameStateView view, CancellationToken ct);
    UniTask<CallDecision> DecideCallAsync(CallContext ctx, CancellationToken ct); // 티츄/큰 티츄
}

// 구현체는 교체 가능 — 룰 엔진은 차이를 모른다
class HumanAgent   : IDecisionAgent { /* UI 입력 대기 */ }
class NetworkAgent : IDecisionAgent { /* 원격 입력 수신 */ }
class AiAgent      : IDecisionAgent { /* ISMCTS + 휴리스틱 */ }

장점

  • 룰 로직 단일화 → AI/사람 간 룰 불일치 원천 차단
  • 헤드리스 self-play로 야간 대량 시뮬레이션·밸런스 측정 가능
  • 난이도는 정책/예산 교체만으로 확장 (엔진 무변경)
  • 리플레이·결정화 시드 고정으로 재현 가능한 디버깅

단점 / 유의

  • 탐색용 순수 데이터 스냅샷 유지 비용(메모리/복사)
  • 스레드 경계에서 Unity 오브젝트 접근 금지 규율 필요
  • 상위 티어 튜닝(휴리스틱 가중치)은 별도 실험 파이프라인 요구
검증 전략: Expert AI 간 self-play 1만 판 이상을 백그라운드로 돌려 (1) 티츄 콜 성공률, (2) 폭탄 평균 사용 시점, (3) 팀 점수 분포를 KPI로 추적한다. 정식 룰 회귀 테스트는 동일 코어 엔진에 대해 별도 단위 테스트 스위트로 상시 실행한다.
향후 확장(보류): 휴리스틱 가중치/리프 평가를 학습 기반(자기대국 강화학습)으로 대체할 여지를 남겨둔다. 단, 프로토타입 단계에서는 비용/리스크를 고려해 ISMCTS + 휴리스틱 조합을 기준선으로 확정한다.

9.2 설계 전제

경량 메시지 유리

턴제라 동기화 대상은 "누가 무엇을 냈는가"뿐. 물리/보간이 없어 이벤트 기반으로 충분하고, 모바일 셀룰러에서도 안정적.

무운영 · 올 C# 1인 최적

Photon Cloud가 서버를 호스팅 → 빌드·배포·모니터링할 인프라가 없음. 룰 엔진을 클라/호스트가 같은 C#로 공유.

권위는 단계적 유예

엄격한 서버 권위는 랭킹 무결성에 이상적이나, 소규모 초기엔 호스트 모델을 수용하고 필요 시 이전(9.4).

호스트 모델의 한계: Photon Cloud 기본은 한 클라이언트가 호스트가 되어 전체 상태(상대 손패 포함)를 보유하므로 엄밀한 랭킹 무결성엔 약점이 있다. 소규모 초기엔 수용(니치 카드게임·소수 유저에서 조직적 치팅 위험 낮음)하되, 랭킹 비중이 커지면 권위 서버로 이전한다.

소규모는 호스트 모델로 시작($0·무운영), 랭킹 비중이 커지면 권위 서버로 이전. 클라이언트는 ITichuTransport 추상화로 교체 영향을 최소화한다.

교체 비용 최소화: 클라이언트는 전송 계층을 ITichuTransport 인터페이스로 추상화한다(9.6). "Photon 호스트 구현"을 "전용 권위 서버(또는 Nakama) 구현"으로 갈아끼워도 게임/UI 코드는 그대로 유지된다.

10 기술 아키텍처 (Unity / C#)

티츄의 코어 가치인 정식 룰의 정확성과 고도화된 AI는 "Unity에 묶이지 않은 순수 C# 룰엔진"에서 출발합니다. 룰과 AI를 헤드리스(headless)로 단위테스트·대량 시뮬레이션할 수 있어야 하고, 연출·네트워크는 그 위에 얇게 얹습니다. 이를 위해 의존 방향이 항상 안쪽(Core)을 향하는 레이어드 아키텍처를 채택하고, asmdef로 레이어를 물리적으로 격리합니다.

10.4 asmdef 기반 레이어 격리

각 레이어를 별도 Assembly Definition으로 분리해 (1) 잘못된 방향의 참조를 컴파일 단계에서 차단하고, (2) 변경 시 해당 어셈블리만 재컴파일되어 이터레이션 속도를 확보합니다. Tichu.Core.asmdef는 참조 목록이 비어 있어야 하며(특히 Unity 모듈 미참조), 이를 CI에서 정적 검사합니다.

장점

  • 아키텍처 규칙을 컴파일러가 강제 (역방향 참조 = 빌드 실패)
  • 부분 재컴파일로 도메인 반복 속도 향상
  • Core를 별도 .NET 프로젝트로 테스트/시뮬레이션 가능
  • 테스트 어셈블리(EditMode/PlayMode) 분리 용이

단점

  • 초기 asmdef 분할·참조 설정 비용
  • 레이어 간 형(type) 노출은 인터페이스/DTO로 의식적 설계 필요
  • 너무 잘게 쪼개면 관리 부담 → 위 6개 단위로 유지

10.7 멀티스레드 / 스레딩 모델

AI 탐색과 직렬화는 계산 비용이 크므로 워커 스레드에서 수행하되, 모든 Unity API(Transform, MonoBehaviour, DoTween 등)는 반드시 메인 스레드에서만 호출합니다. Core가 순수 C#이라는 점이 여기서 결정적 이점이 됩니다 — 룰/AI 연산은 Unity 컨텍스트 밖에서 안전하게 스레드풀로 보낼 수 있습니다.

  1. 입력/턴 시작 시점: 메인 스레드에서 GameState 스냅샷을 만든다 (불변·복사본).
  2. AI 계산: await UniTask.RunOnThreadPool(() => ai.Decide(snapshot)) 로 워커에서 탐색.
  3. 결과 회수: UniTask가 자동으로 메인 스레드 컨텍스트로 복귀시켜 준다.
  4. 연출 적용: 복귀 후 DoTween/UniRx로 UI 갱신 (모두 메인 스레드).
주의: 워커로 넘기는 GameState는 반드시 불변 스냅샷이어야 한다. 메인 스레드가 동시에 수정 가능한 가변 상태를 공유하면 경합(race)이 발생한다. AI는 입력 스냅샷만 읽고 "결정(Command)"만 반환하도록 설계한다.
절대 금지: 워커 스레드에서 Transform·GameObject·DoTween 호출, Time.deltaTime 접근, MonoBehaviour 필드 변경. 위반 시 비결정적 크래시 또는 무음 데이터 손상으로 이어진다. 메인 스레드 복귀가 필요하면 UniTask.SwitchToMainThread()를 사용한다.

10.8 정리

  • 안쪽으로의 의존을 asmdef로 강제 → 룰/AI를 화면·네트워크 없이 테스트·시뮬레이션.
  • UniTask(흐름·await·취소) / UniRx(상태→UI) / DoTween(연출) 역할을 명확히 분리, 모두 바깥 레이어에 한정.
  • 포트-어댑터로 온라인/싱글(AI)/로컬을 동일 GameFlow에서 교체 가능.
  • 스냅샷 기반 워커 스레딩으로 AI 성능 확보, Unity API는 메인 스레드 고정.
  • Shop/Ads/Avatar(BM)는 인터페이스만 선언해 두는 구조로 보류.

11 데이터 모델 & 룰 엔진

티츄의 정확성은 전적으로 룰 엔진에 달려 있습니다. 본 섹션은 게임 상태를 표현하는 핵심 C# 타입과, 그 위에서 동작하는 결정적(deterministic)·헤드리스(headless) 룰 엔진을 정의합니다. 룰 엔진은 Unity API에 의존하지 않으며, 서버 권위 검증 · 싱글 AI 시뮬레이션 · 리플레이 재생이 모두 동일한 코드를 공유합니다. 이것이 "정식 룰의 정확성"과 "고도화된 AI"라는 두 핵심 가치를 동시에 떠받치는 토대입니다.

룰 엔진(Tichu.Core 어셈블리)은 순수 C#으로 작성합니다. UnityEngine 참조 금지, Random은 시드 주입식만 허용, 부동소수점은 봉황 0.5 비교 한 군데로 격리합니다. 이렇게 해야 서버와 클라이언트, AI가 비트 단위로 동일한 판정을 내립니다.

11.1 핵심 C# 타입

카드 표현 (Card Representation)

카드는 struct(값 타입)로 설계해 할당 압박과 GC를 줄입니다. 손패는 List<Card>가 아니라 정렬·비교가 빈번하므로, 비교용 정수 키(SortValue)를 함께 들고 다닙니다. 특수 카드는 Suit.Special + SpecialKind 조합으로 식별합니다.

// 문양: 옥(Jade/녹), 검(Sword/흑), 탑(Pagoda/청), 별(Star/적), 그리고 특수
public enum Suit { Jade, Sword, Pagoda, Star, Special }

// 특수 4장
public enum SpecialKind { None, Mahjong, Dog, Phoenix, Dragon }

// 카드 1장 (값 타입)
public readonly struct Card : IEquatable<Card>
{
    public readonly int Rank;            // 일반: 2~14(A=14). 특수: Mahjong=1, 나머지는 비교시 별도 처리
    public readonly Suit Suit;           // 일반 4문양 또는 Special
    public readonly SpecialKind Special; // 특수 카드 종류 (일반 카드는 None)
    public readonly int Points;          // K=10, 10=10, 5=5, Dragon=+25, Phoenix=-25, 그 외 0

    public bool IsSpecial => Suit == Suit.Special;
    public bool Equals(Card o) => Rank == o.Rank && Suit == o.Suit && Special == o.Special;
}
봉황의 "단독 시 직전 값 +0.5" 규칙 때문에 Card.Rank(정수)와 조합 비교용 값을 분리합니다. 봉황의 비교값은 카드 자체가 아니라 해당 트릭의 문맥에서 결정되므로 Combination.Rank(아래)에서 다룹니다.

조합 (Combination)

public enum CombinationType
{
    Invalid,          // 비합법 조합 (판별 실패)
    Single,           // 단독 1장
    Pair,             // 페어 (같은 랭크 2)
    Triple,           // 트리플 (같은 랭크 3)
    FullHouse,        // 풀하우스 (트리플 + 페어)
    Straight,         // 스트레이트 (연속 5장 이상)
    ConsecutivePairs, // 연속 페어/계단 (2쌍 이상 연속)
    FourBomb,         // 포카드 폭탄
    StraightFlushBomb // 스트레이트 플러시 폭탄 (같은 문양 연속 5장+)
}

public sealed class Combination
{
    public CombinationType Type;
    public IReadOnlyList<Card> Cards;  // 실제 낸 카드(봉황 포함)
    public int Length;                 // 카드 장수 (스트레이트/계단 비교의 일부)
    public float Rank;                 // 비교 기준값. 봉황 단독만 .5 사용, 그 외 정수
    public int PointsInPlay;           // 이 조합이 머금은 점수 (트릭 정산용)

    public bool IsBomb => Type == CombinationType.FourBomb
                       || Type == CombinationType.StraightFlushBomb;
}
Rank를 float로 둔 이유는 오직 봉황 단독(직전+0.5, 선 첫 단독=1.5) 한 가지입니다. 그 외 모든 조합은 정수값을 갖습니다. 비교 시 동일 Type·Length 내에서 Rank만 비교하면 되도록 설계해, 비교 로직을 단순·검증 가능하게 유지합니다.

트릭 · 좌석 · 게임 상태

11.2 핵심 클래스 관계

핵심 클래스 관계도. GameState가 루트이며 좌석 · 트릭 · 점수판을 집약한다. Combination은 룰 엔진이 손패(Card[])에서 파생시키는 1차 시민이다.

11.3 룰 엔진 책임

룰 엔진은 IRuleEngine 인터페이스 하나로 노출되며, 모든 메서드는 GameState를 입력으로 받아 순수 함수적으로 결과를 산출합니다(상태 변경은 Apply 계열 메서드만 수행).

① 조합 판별 / 검증

Recognize(Card[]) → Combination. 임의 카드 묶음이 합법 조합인지 식별하고 타입·장수·비교값을 산출. 봉황 와일드 채움 포함. 실패 시 Invalid.

② 비교 (이김 판정)

Beats(candidate, current) → bool. 추종 타입·장수 일치 + 더 높은 값일 때만 true. 폭탄 계층(포카드 < 스플), 봉황 단독 0.5, 용 최강 단독을 모두 처리.

③ 합법 수 생성

LegalMoves(state, seat) → Combination[]. 손패에서 현재 트릭을 받을 수 있는 모든 합법 조합 + 패스. AI 탐색과 클라이언트 하이라이트가 공유.

④ 점수 계산

ScoreRound(state) → RoundResult. 원-투/일반 정산, 티츄·큰 티츄 가감, 마지막 1인 손패·트릭 양도까지 정식 룰대로 산출.

⑤ 결정적 / 헤드리스

Unity 비의존, 시드 주입 셔플. 동일 입력 → 동일 출력 보장. 서버 권위 검증과 AI 몬테카를로 시뮬, 리플레이가 한 코드를 공유.

⑥ 상태 전이

Apply(state, action) → 새 GameState 또는 거부 사유. 페이즈 FSM 가드와 소원 강제·아웃 처리·트릭 회수를 단일 진입점으로 통합.

public interface IRuleEngine
{
    Combination Recognize(ReadOnlySpan<Card> cards, TrickContext ctx);
    bool Beats(Combination candidate, Trick current);
    IReadOnlyList<Combination> LegalMoves(GameState s, int seat);
    bool IsLegal(GameState s, int seat, Combination move, out string reason);
    ApplyResult Apply(GameState s, GameAction action); // 결정적 전이
    RoundResult ScoreRound(GameState s);
}
한 가지 진실(single source of truth). 클라이언트는 UX(낼 수 있는 카드 하이라이트)를 위해 LegalMoves·IsLegal을 호출하지만, 최종 합법성은 서버가 동일 엔진으로 재검증합니다. 클라이언트 판정은 편의이지 권위가 아니므로 치팅에 강건합니다.

11.4 비교 규칙 우선순위

비교는 룰 엔진에서 가장 버그가 잦은 부분이라 결정 트리를 명시합니다. 폭탄은 자기 차례가 아니어도 끼어들 수 있으므로 비교는 항상 "현재 트릭 Top vs 후보" 2자 간 판정으로 환원됩니다.

후보 \ Top비폭탄 (동일 타입)포카드 폭탄스트레이트 플러시
비폭탄타입·장수 같고 값↑ 이면 이김패배패배
포카드 폭탄무조건 이김랭크↑ 만 이김패배
스트레이트 플러시무조건 이김무조건 이김더 길거나(우선) 같은 길이면 값↑ 만 이김

단독 비교의 특수값

  • 용(Dragon) = 모든 단독을 이기는 최강값(A=14보다 위). 단독으로만 출전.
  • 봉황(Phoenix) 단독 = 직전 단독값 + 0.5 (용은 못 이김). 선 첫 단독으로 낼 땐 1.5.
  • 마작(Mahjong) = 값 1, 가장 낮음. 스트레이트의 최하단(1~A)에만 끼움.

봉황 와일드 채움

  • 페어·트리플·풀하우스·스트레이트·연속페어에서 빈 자리 1장을 대체.
  • 폭탄(포카드·스플)에는 절대 사용 불가.
  • Recognize는 봉황을 가장 트릭에 유리한 값으로 자동 배정(예: 추종 시 정확히 받을 수 있는 값).

11.5 라운드 페이즈 상태머신 (FSM)

한 판의 진행은 명시적 FSM으로 모델링합니다. 각 전이는 Apply의 가드(guard)로 강제되어, 잘못된 순서의 액션(예: 교환 전에 카드 내기)은 거부됩니다.

라운드 페이즈 상태머신. Play는 트릭 단위 자기 루프를 돌며, 3인 아웃 또는 원-투 즉시종료 조건에서 Scoring으로 전이한다. RoundEnd에서 목표 점수 미달이면 새 판(Deal8)으로 순환한다.

11.6 한 판 진행 (정산 요지)

  1. 8장 배분 → 큰 티츄(±200) 콜 결정(8장만 본 상태).
  2. 나머지 6장 배분(총 14장).
  3. 카드 교환: 왼·파트너·오른에게 1장씩 전달 후 동시 공개.
  4. 마작(1) 보유자가 첫 선. 이후 작은 티츄(±100)는 자신의 첫 카드 내기 전까지 콜 가능.
  5. 트릭 반복: 추종(같은 타입·장수·값↑) 또는 폭탄 또는 패스. 모두 패스 시 마지막 낸 사람이 회수하고 다음 선.
  6. 종료: 3명 아웃 시 종료. 단 한 팀이 1·2등으로 먼저 나가면(원-투) 즉시 종료.
정산 항목규칙
원-투(One-Two)해당 팀 +200, 그 판의 카드 점수는 무시.
일반 종료마지막 1인: ⒜ 손에 남은 카드 점수를 상대팀에 ⒝ 자신이 딴 트릭은 1등에게 양도 → 팀별 카드점수 합산(덱 총점 100).
티츄 / 큰 티츄성공 시 +100 / +200, 실패 시 -100 / -200 (정산에 별도 가감).
용 트릭용으로 이긴 트릭은 그 전체를 상대팀 한 명에게 양도(점수가 상대에게 감).
게임 종료한 팀이 목표 점수(기본 1000, 설정 가능) 도달 시.

11.7 엣지케이스 처리

소원(Wish) 강제 검증

마작과 함께 소원 숫자가 지정되면 GameState.Wish에 보관. 이후 각 플레이어의 LegalMoves는 "그 숫자를 합법적으로 포함해 낼 수 있는 경우, 반드시 포함한 수만" 반환합니다. 즉 소원 충족 가능한 조합이 손에 있으면 패스·다른 수를 거부. 누군가 그 숫자를 내 소원이 충족되면 Wish = null로 해제. 폭탄으로 끼어들 때도 소원 검사가 적용됩니다.

봉황 와일드

Recognize에서 봉황은 후보 조합을 만족시키는 일반 카드 값으로 자동 치환. 추종 트릭에선 "현재 Top을 정확히 이기는 최소값"으로 배정해 손패 손실을 줄입니다. 폭탄 판별 경로에서는 봉황을 즉시 제외(폭탄 불가). 점수는 항상 -25 고정.

개(Dog) 처리

선·단독으로만 합법. 트릭을 형성하지 않고 즉시 선을 파트너에게 이전. 파트너가 이미 아웃이면 그 다음 사람으로. Apply에서 Trick을 생성하지 않고 Turn만 갱신하는 특수 분기로 처리합니다.

봉황 단독 비교 / 용 양도

봉황 단독 비교값은 카드가 아니라 트릭 문맥(TrickContext.PrevSingleRank)에서 계산: 직전+0.5, 선 첫 수는 1.5. 용이 트릭을 이기면 Trick.WonByDragon = true로 표시, 정산 시 상대팀 한 명에게 강제 양도.

부동소수점 격리. 봉황 0.5 비교 외에는 절대 float를 비교에 쓰지 않습니다. 결정성이 깨지면 서버-클라이언트 desync와 리플레이 불일치가 발생하므로, 봉황 단독 비교조차 정수×2 스케일(예: 직전값×2 + 1)로 환산해 정수 비교로 처리하는 것을 권장합니다.

11.8 테스트 전략

  • 골든 케이스 테이블: 정식 룰 문서의 모든 비교 사례(봉황 단독, 폭탄 계층, 스플 길이 우선)를 데이터 드리븐 NUnit 테스트로 고정.
  • 프로퍼티 기반 테스트: LegalMoves가 반환한 모든 수는 IsLegal을 통과해야 한다는 불변식 검증(랜덤 시드 손패 수천 건).
  • 정산 회귀: 원-투 / 일반 / 티츄 실패 조합을 명시적 시나리오로 고정해 점수 합이 항상 보존(덱 100점 + 티츄 가감)되는지 검증.
  • 결정성 검증: 동일 시드·동일 액션 로그 재생 시 GameState 해시가 비트 단위로 일치하는지 확인(서버·AI 공유 전제).

12 개발 로드맵 & 우선순위

티츄는 "정식 룰의 정확성"과 "4인 화면의 가독성"이 게임성의 근간이며, 온라인은 그 위에 얹히는 레이어입니다. 따라서 로드맵은 코어 룰엔진을 가장 먼저 검증하고(헤드리스 테스트), 손맛(인게임 UI/애니)으로 재미를 검증한 뒤, AI를 고도화하고, 마지막에 온라인을 붙이는 순서로 설계했습니다. 각 Phase는 명확한 산출물(Deliverable)과 완료 기준(Definition of Done, DoD)을 가지며, 다음 Phase 착수의 게이트 역할을 합니다.

우선순위는 사용자 재검토 후 확정. 아래 단계 구성·기간·DoD는 프로토타입 검증을 위한 제안안입니다. 실제 착수 전 이해관계자 재검토를 거쳐 범위와 순서를 확정합니다(특히 Phase 3 네트워킹 선택안과 Phase 4 보류 항목).

12.1 우선순위 근거

1순위 룰 정확성

봄(Bomb), 위상권(Phoenix), 용(Dragon), 개(Dog), 연속 형태(스트레이트/풀하우스/연속 페어), 트릭/라운드/점수 정산 등 엣지케이스가 많습니다. 코어가 틀리면 모든 상위 레이어가 무의미하므로 최우선.

1순위 가독성·손맛

4인 동시 정보(손패/콤보/패스/티츄 콜)를 한 화면에서 읽히게 하는 것이 제품의 차별점. MVP 단계에서 조기 검증.

2순위 AI 고도화

싱글 모드 핵심 가치이자 온라인 봇 대체재. 룰엔진이 확정된 뒤 ISMCTS로 강화.

온라인(Phase 3)은 가치가 높지만 리스크·비용이 크고 코어 품질에 의존적입니다. "재미가 검증된 코어 + 믿을 만한 AI"가 선행되어야 온라인 투자 대비 효과가 극대화되므로 후순위로 배치했습니다. BM(상점/광고/아바타)은 Phase 4 보류 항목으로 구조만 확보합니다.

12.2 단계별 타임라인 (마일스톤 도식)

Phase 0~4 마일스톤과 게이트(G0~G3). 실선은 필수 순차 의존, 점선은 보류/조건부 진행.

12.3 Phase별 산출물 & 완료 기준(DoD)

Phase 0 코어 룰엔진 + 헤드리스 단위테스트 엔진 비의존

목표: Unity·렌더링과 완전히 분리된 순수 C# 룰엔진(Tichu.Core) 구축. 결정적(deterministic)·테스트 가능하게 설계.

산출물

  • 도메인 모델: Card, Combination(싱글/페어/트리플/풀하우스/스트레이트/연속페어/봄), 특수카드(Dragon·Phoenix·Dog·Mahjong) 규칙
  • 게임 상태기계: 카드교환(3장) → 트릭 진행 → 패스/봄 인터럽트 → 라운드/판 종료 → 점수 정산(1-2 피니시, 티츄/그랜드티츄 콜 ±100/±200)
  • 합법수(legal move) 생성기 + 규칙 검증기, 시드 기반 셔플(재현 가능)
  • 헤드리스 시뮬레이터(CLI)로 수만 판 자동 대국(AI 무관, 랜덤/규칙봇)
  • NUnit 단위테스트 스위트 + 엣지케이스 시나리오, CI 연동

완료 기준(DoD)

  • 룰 단위테스트 통과율 100%, 핵심 규칙 코드 커버리지 ≥ 90%
  • 10만 판 헤드리스 시뮬레이션에서 불법 상태/예외 0건, 점수 합 정합성(제로섬·총점 검증) 100%
  • 모든 특수카드·봄·콜 시나리오가 명세 표(섹션 룰 문서)와 1:1 매핑되어 검증됨

Phase 1 MVP: 싱글(로컬) + 기본 AI + 인게임 UI/애니(손맛 검증)

목표: 한 디바이스에서 vs AI 한 판을 끝까지 즐길 수 있고, "재미있다"가 검증되는 최소 제품.

산출물

  • 룰엔진 위에 얹는 프레젠테이션 레이어: 4인 테이블 레이아웃, 내 손패 정렬/선택, 콤보 미리보기, 콜/패스 UI
  • UniRx 기반 상태→UI 바인딩, DoTween 카드 딜/플레이/회수 애니메이션, 타격감(하이라이트·사운드 훅)
  • 기본 AI(룰/휴리스틱): 합법수 중 휴리스틱 선택, 카드교환·티츄 콜 단순 정책
  • 로컬 1인 + AI 3인 풀 게임 루프, 일시정지/재시작, 결과 화면

완료 기준(DoD)

  • vs AI 한 판(1000점 기준) 크래시 없이 완주, 모든 룰 인터랙션이 UI에서 정상 동작
  • 중저가 안드로이드 기준 60fps 유지, 애니메이션 끊김/입력 지연 체감 없음
  • 내부 플레이테스트에서 "한 판 더" 의향 등 손맛 정성 지표 합격(가독성 체크리스트 통과)

Phase 2 AI 고도화 (ISMCTS / 난이도)

목표: 불완전정보(상대 손패 비공개) 환경에 적합한 ISMCTS(Information Set MCTS) 기반 AI와 난이도 체계 확립.

산출물

  • ISMCTS 엔진(결정화 determinization + UCT), 파트너 협력/콜 의사결정 통합
  • 난이도 티어(쉬움·보통·어려움): 시뮬레이션 budget·휴리스틱 가중치로 조절
  • UniTask 기반 비동기 사고(메인 스레드 비차단), 모바일 thinking time 예산 관리
  • 헤드리스 AI 대 AI 벤치마크 하니스(난이도별 승률·평균 사고시간 리포트)

완료 기준(DoD)

  • '어려움' AI가 기본 휴리스틱 봇 대비 통계적으로 유의한 승률 우위(다수 판 기준)
  • 1수당 사고시간 모바일 목표 예산 내(예: 평균 ≤ 1초, 최악 ≤ 2초), 프레임 드랍 없음
  • 난이도 간 체감 격차가 플레이테스트로 확인되고, 결정적 시드로 재현 가능

Phase 3 온라인 (랭킹전 / 친구방, 권위 서버, 매치메이킹, 재접속)

목표: 검증된 룰엔진을 서버 권위(authoritative)로 재사용하여 치팅 방지·일관성을 확보한 실시간 4인 온라인.

산출물

  • 선택된 네트워킹 스택(섹션 네트워킹 참조) 위 게임 서버: 서버가 룰·셔플·상태 권위 보유, 클라는 입력 전송·상태 수신
  • 랭킹전 매치메이킹(레이팅 기반), 친구방(코드/초대), 방 로비
  • 재접속(reconnect): 세션 토큰·스냅샷 복구, 이탈 시 자리 AI 대체 정책
  • 지연/끊김 처리: 타임아웃·자동 패스, 서버 검증 거부 시 클라 롤백

완료 기준(DoD)

  • 실제 4인 원격 매칭으로 한 판 완주, 서버-클라 상태 불일치 0건
  • 강제 종료/네트워크 단절 후 재접속 시 정확한 보드 상태로 복귀(스냅샷 검증)
  • 매치메이킹 평균 대기시간·서버 지연(p95) 목표치 충족, 부하 테스트 통과

Phase 4 (보류) 계정 / 리더보드 / 상점 / 광고

목표: 구조만 선반영하고 실제 활성화는 별도 의사결정. 본 로드맵에서는 인터페이스·확장 지점만 확보.

산출물(구조만)

  • 계정/인증 추상화(게스트→정식 전환 지점), 프로필·시즌 리더보드 스키마
  • 상점/아바타/광고는 어댑터 인터페이스로 격리(미구현 스텁), BM 로직과 게임플레이 분리

완료 기준(DoD)

  • 핵심 게임 코드가 BM 모듈에 의존하지 않음(역의존 only), 후속 활성화가 비파괴적
  • 활성화 범위·일정은 사용자 재검토 후 확정

12.5 접근 방식의 장단점

장점

  • 코어 선검증으로 상위 레이어의 룰 버그 리스크 제거(엔진 재사용: 싱글·AI·서버 동일 코드)
  • 헤드리스 테스트/시뮬으로 회귀를 CI에서 자동 차단, AI 강화도 같은 하니스로 측정
  • MVP에서 재미를 조기 검증 → 온라인 투자(고비용)를 근거 기반으로 결정
  • 서버 권위 구조가 치팅 방지·상태 일관성·재접속을 자연스럽게 지원

단점 / 리스크

  • 온라인 가시 성과가 후반에 나와 초기 외부 데모 임팩트가 약할 수 있음
  • Phase 0 선투자 비용(설계 규율 필요), 일정 압박 시 단축 유혹
  • ISMCTS 모바일 성능 튜닝 난이도(사고시간 vs 강도 트레이드오프)
  • 네트워킹 스택 선택 결과에 Phase 3 일정이 종속
권장 진행: Phase 0·1을 묶어 "재미가 검증된 정확한 코어"를 먼저 확보하고, 그 결과를 게이트 삼아 Phase 2(AI)·3(온라인) 투자 규모를 재산정합니다. Phase 4는 명시적 보류로 두고 구조적 확장 지점만 유지합니다.

13 리스크 & 결정 필요 사항

본 섹션은 프로토타입 착수 전에 의사결정권자의 승인이 필요한 항목과, 프로젝트 전반에 걸쳐 추적·관리해야 할 핵심 리스크를 정리한다. 각 항목은 영향도와 발생 가능성으로 등급화하고 구체적 완화책을 명시했다. 마지막에 검토부터 Phase 0 착수까지의 진행 절차를 안내한다.

13.1 승인이 필요한 결정 항목 (Decision Gate)

아래 4개 항목은 상호 의존성이 있어 Phase 0 착수 이전에 일괄 확정되어야 한다. 미결 시 아키텍처·예산·일정이 연쇄적으로 흔들린다.

승인 요청 — 4개 항목 (응답 기한: 검토 회의 +3영업일)
  • ① 네트워킹 방안 선택 — 9장 권장안(전용 권위 서버 + WebSocket 기반 상태 동기화)을 채택할지 확정. 대안은 P2P/릴레이 또는 매니지드 리얼타임 SDK(Photon/Nakama). 권장: 9장 권위 서버안(룰 정확성·치팅 방지 우선).
  • ② 인게임 화면 방향 — 4인 좌석 배치·핸드 가독성을 위한 가로(Landscape) 확정 여부. 권장: 가로 고정. 세로 병행 시 레이아웃 2벌·QA 비용 약 1.6배 증가.
  • ③ 백엔드 호스팅/비용 — 클라우드 사업자(예: AWS/GCP) 및 초기 인스턴스 등급, 동시접속(CCU) 가정, 월 예산 상한. 매니지드 SDK 채택 시 과금 모델(MAU/CCU)도 함께 결정.
  • ④ AI 연산 예산 — 모바일 1턴당 허용 연산 시간(목표 ≤ 300ms) 및 난이도별 탐색 깊이. 서버측 AI 오프로딩 허용 여부(허용 시 ③ 비용에 합산).

결정 게이트: 4개 승인 항목이 모두 확정되어야 Phase 0가 착수된다. ①·②는 아키텍처를, ③·④는 예산을 결정한다.

봉황(Phoenix)·용(Dragon)·소원(Mahjong wish)·1-2위 정산(원-투)		 상 중 룰 엔진을 순수 결정함수로 분리하고 엣지케이스 단위테스트 세트 구축(봉황 0.5 가치/스트리트 내 보간, 용 단독 최강·상대팀 점수 양도, 소원 강제 콜, 원-투 200점 즉시 종료). 정식 룰 표를 단일 출처(SSOT)로 고정.
R4. 치팅 방지
핸드 노출, 패킷 변조, 클라이언트 룰 우회		 상 중 권위 서버에서만 룰 판정·핸드 보유(클라이언트는 본인 핸드만 수신). 모든 액션 서버 검증. 부정 패턴 로깅·랭킹 무효 처리. (①에서 권위 서버안 채택이 전제.)
R5. 아트 에셋 일정
카드/문양(옥·검·탑·별)·UI 키트 납기 지연		 중 중 프로토타입은 플레이스홀더(도형+텍스트)로 선개발. 에셋 의존 작업을 임계경로에서 분리. 문양 색 코드·해상도 가이드 사전 확정으로 재작업 방지.
R3·R4는 최우선 가치(룰 정확성·고도화 AI)와 직결되므로, 영향도 산정 시 가중치를 높게 두었다. 두 항목 모두 ① 네트워킹 결정에 종속된다.

13.3 리스크 매트릭스 (영향 × 발생 가능성)

우상단(붉은 영역)에 위치한 R1·R3·R4는 최우선 대응 대상이며, R2는 발생 가능성이 높아 기술 검증(PoC)을 선행한다.

13.4 권장 결정에 대한 장단점

장점

  • 권위 서버안(①) 채택 시 R3·R4(룰·치팅)를 구조적으로 차단 — 최우선 가치 직접 달성.
  • 가로 고정(②)으로 4인 좌석·핸드 가독성 극대화, 레이아웃·QA 비용 절감.
  • AI 시간예산(④) 명문화로 저사양 대응과 일정 예측이 가능.

단점

  • 권위 서버는 초기 호스팅 비용·운영 부담(R1)이 P2P 대비 큼.
  • 가로 고정은 일부 캐주얼 세로 사용자 UX를 일부 제약.
  • 서버 AI 오프로딩 허용 시 ③ 비용이 추가 증가.
